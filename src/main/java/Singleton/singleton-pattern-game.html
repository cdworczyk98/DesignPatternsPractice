Main.java — The Problem is Already HereJava
public class Main {

        GameConfig audioConfig = new GameConfig();
        audioConfig.setVolume(50); // player turned it down

The Seed of the Problem

Every class that needs config creates its own new GameConfig(). Each instance is completely independent. Changes in one are invisible to all others. Click ② to see this fully break.

> STEP_02

NEW REQUIREMENTS THAT BREAK IT

The game grows. Now AudioSystem, GraphicsSystem, InputManager, NetworkManager, and the SettingsMenu all need access to the same config. Each creates its own instance. They all diverge immediately.

Every system creates its own config — divergence guaranteedJava
public class AudioSystem {
    private GameConfig config = new GameConfig(); // instance A
    public void play() { /* uses config.getVolume() */ }
}

public class GraphicsSystem {
    private GameConfig config = new GameConfig(); // instance B — different!
    public void render() { /* uses config.getBrightness() */ }
}

public class SettingsMenu {
    private GameConfig config = new GameConfig(); // instance C — also different!

    public void onVolumeSlider(int v) {
        config.setVolume(v); // writes to C only
        // AudioSystem's config (A) never gets this update.
        // Player changes volume — game audio doesn't respond.
    }
}
⚠ THE BUG

Player opens the Settings Menu, slides volume to 20. Closes the menu. Audio is still at 70. The menu wrote to its own config copy, not the AudioSystem's copy. This is a real, hard-to-diagnose bug.

Problem

State Divergence

Five config instances drift apart immediately. There is no "current" config — there are five competing versions of truth.

Problem

Wasted Memory

N systems = N identical config objects in memory. For a lightweight config this is trivial — for a database connection pool it is catastrophic.

Problem

No Coordination

A "save to disk" call on one instance doesn't include changes made to other instances. Saves are incomplete.

Problem

Constructor Pollution

Every class that needs config must receive it via constructor injection or create its own — leading to messy, deeply coupled constructors.

The Core Insight

Some objects should only ever exist once. A game can't have two sets of settings. A printer has one print queue. A system has one clock. The Singleton Pattern guarantees that a class has exactly one instance and provides a global access point to it.

> STEP_03

DEEP DIVE

Singleton is the most controversial GoF pattern. It's simple to implement badly and subtle to implement correctly. This panel covers all four implementations, thread safety in depth, and when NOT to use it.

Diagram 1 — Singleton Structure

There are only three moving parts. The private constructor prevents outside instantiation. The static field holds the one instance. The static getInstance() method is the single access point.

Diagram 2 — The Thread Race Condition

The naive implementation has a critical flaw: two threads can both see instance == null simultaneously and both create a new object. The result is two singletons — the exact problem we were trying to prevent.

Diagram 3 — Double-Checked Locking Fix

The solution: check null twice — once without the lock (fast path for the 99.9% case), once inside the lock (safety for the race condition). The volatile keyword prevents the JVM from partially constructing the object before assigning the reference.

> FOUR_IMPLEMENTATIONS
Naive
Lazy, NOT thread-safe

Checks null and creates if absent. Works in single-threaded apps. Race condition under concurrency. Never use in production.

Single thread only
Eager
Thread-safe, always initialised

Instance created when class loads. Thread-safe by JVM guarantee. Downside: initialised even if never used (wasted resources if expensive).

Safe but wasteful
DCL
Double-Checked Locking

Lazy + thread-safe. Checks null twice — once without lock, once inside. Requires volatile. The classic production approach.

Production ready
Holder
Bill Pugh / Static inner class

Uses class-loading guarantee. Lazy, thread-safe, no synchronized overhead. Considered the cleanest Java approach by most experts.

Best practice
> WHEN_NOT_TO_USE
  1. When you need testability Singletons are notoriously hard to unit test. A test that mutates the singleton pollutes every subsequent test. Solution: inject the config as a dependency instead of calling getInstance() directly. The singleton can still exist — just don't call it from inside your classes.
  2. When you need multiple environments A singleton config can't be "dev config" and "prod config" simultaneously. If you need environment-specific instances, a factory or dependency injection container is more appropriate.
  3. When the class has mutable state that's frequently written A shared mutable singleton is a concurrency hazard. If many threads write to it constantly, synchronisation overhead kills performance. Consider immutable config objects or a message-passing design instead.
  4. When you're reaching for it just to avoid passing arguments "Global variable with extra steps" is a real anti-pattern. If you're using Singleton purely to avoid threading config through constructors, dependency injection (Spring, Guice) solves the same problem with far less risk.
The Classic Tradeoff

Singleton solves a real problem (one shared state, global access) at a real cost (hidden coupling, test difficulty, concurrency risk). Use it deliberately — not as a default for anything you want globally accessible.

> STEP_04

REFACTORED — ALL FOUR IMPLEMENTATIONS

All four Singleton variants on the same GameConfig class. Compare the trade-offs side by side.

Variant 1 — Naive (DO NOT USE in multithreaded code)

Shown for educational purposes only. Race condition exists between the null check and object creation.

GameConfig.java — Naive Lazy SingletonJava
public class GameConfig {

    // Step 1: static field holds the one instance
    private static GameConfig instance = null;

    // Step 2: private constructor — no outside 'new GameConfig()'
    private GameConfig() {
        // initialise defaults
    }

    // Step 3: public static access point
    public static GameConfig getInstance() {
        if (instance == null) {               // ← RACE CONDITION HERE
            instance = new GameConfig();    // two threads can both reach this
        }
        return instance;
    }

    // ... fields and methods unchanged ...
}
Variant 2 — Eager Initialisation

Thread-safe by JVM class-loading guarantee. Simple and reliable. Only downside: always initialised at class load time.

GameConfig.java — Eager SingletonJava
public class GameConfig {

    // Created when the class is first loaded by the JVM.
    // The JVM guarantees static initialisers run exactly once.
    // Thread-safe with zero synchronisation overhead.
    private static final GameConfig INSTANCE = new GameConfig();

    private GameConfig() {}

    public static GameConfig getInstance() {
        return INSTANCE; // always ready, no null check needed
    }

    // ... fields and methods ...
}
Variant 3 — Double-Checked Locking (DCL)

Lazy + thread-safe. The volatile keyword is mandatory — without it, the JVM may publish a partially-constructed object reference.

GameConfig.java — DCL SingletonJava
public class GameConfig {

    // volatile: prevents JVM instruction reordering.
    // Without volatile, another thread could see a non-null
    // but not-yet-fully-constructed instance.
    private static volatile GameConfig instance;

    private GameConfig() {}

    public static GameConfig getInstance() {
        if (instance == null) {                  // 1st check — no lock (fast path)
            synchronized (GameConfig.class) {   // acquire lock
                if (instance == null) {          // 2nd check — inside lock (safe)
                    instance = new GameConfig();
                }
            }
        }
        return instance;
    }

    // ... fields and methods ...
}
Variant 4 — Static Holder (Bill Pugh) — Recommended

Uses the JVM's class-loading guarantee on an inner class. Lazy, thread-safe, zero synchronisation, no volatile needed. This is the cleanest Java Singleton.

GameConfig.java — Static Holder SingletonJava
public class GameConfig {

    private int    volume     = 70;
    private int    brightness = 80;
    private String difficulty = "NORMAL";
    private String resolution = "1920x1080";

    // Private constructor — enforces single instantiation
    private GameConfig() {
        System.out.println("[GameConfig] initialised once");
    }

    // The inner class is NOT loaded until getInstance() is called.
    // When it IS loaded, the JVM guarantees INSTANCE is created
    // exactly once, thread-safely, with no synchronisation overhead.
    private static class Holder {
        static final GameConfig INSTANCE = new GameConfig();
    }

    public static GameConfig getInstance() {
        return Holder.INSTANCE;
    }

    // Getters and setters
    public int    getVolume()            { return volume; }
    public void   setVolume(int v)        { volume = v; }
    public int    getBrightness()         { return brightness; }
    public void   setBrightness(int b)     { brightness = b; }
    public String getDifficulty()          { return difficulty; }
    public void   setDifficulty(String d)  { difficulty = d; }
    public String getResolution()           { return resolution; }
    public void   setResolution(String r)  { resolution = r; }

    @Override
    public String toString() {
        return String.format("[CONFIG] vol:%d bright:%d diff:%s res:%s",
            volume, brightness, difficulty, resolution);
    }
}
Main.java — Proving One InstanceJava
public class Main {
    public static void main(String[] args) {

        // Every system uses getInstance() — same object every time
        GameConfig audio    = GameConfig.getInstance();
        GameConfig graphics = GameConfig.getInstance();
        GameConfig settings = GameConfig.getInstance();

        audio.setVolume(50); // player turns down volume via SettingsMenu

        System.out.println(graphics.getVolume()); // 50 — not 70! ✓
        System.out.println(settings.getVolume()); // 50 ✓

        // Prove it's the SAME object:
        System.out.println(audio == graphics); // true ✓
        System.out.println(audio == settings); // true ✓

        // Output:
        // [GameConfig] initialised once  ← constructor called ONCE
        // 50
        // 50
        // true
        // true
    }
}
> STEP_05

PRACTICE CHALLENGES

Eight challenges — from basic implementation to thread-safety and pattern limits. All build on the Static Holder GameConfig from panel ④.

LEVEL 1Challenge 1

Prove the Single Instance in Code

Write a ProofTest class that calls GameConfig.getInstance() from three separate methods and verifies all three references point to the same object using ==. Then call setVolume() on one reference and verify the change is visible on all three.

Goal

Cement the core guarantee: one instance, shared state, all references see the same data.

LEVEL 1Challenge 2

Add Input Validation to Setters

Add guards: setVolume() should clamp input to [0, 100] and throw IllegalArgumentException for values outside that range. setDifficulty() should only accept "EASY", "NORMAL", or "HARD". Since there is only one instance, invalid state would corrupt the entire game — validate aggressively.

Think About

With a singleton, invalid state isn't just a local problem — it poisons every system that reads from it. Validation at the setter is critical.

LEVEL 2Challenge 3

Thread-Safety Stress Test

Write a stress test using ExecutorService with 100 threads. Each thread calls GameConfig.getInstance() and stores the reference. After all threads finish, assert that all 100 references are identical. Also add a counter to the private constructor and assert it was called exactly once.

Think About

Try this with the Naive variant first (it may or may not fail depending on timing — that's the point). Then switch to Static Holder and confirm it always passes.

LEVEL 2Challenge 4

Save and Load Config

Add saveToFile(String path) and loadFromFile(String path) methods to GameConfig using java.util.Properties. When the game starts, call loadFromFile("config.properties") on the singleton. Verify that settings persist across simulated restarts.

Think About

Should loading happen in the constructor, in a separate init() method, or lazily? What happens if the file doesn't exist yet? What if it's corrupted?

LEVEL 2Challenge 5

Observer Integration — Notify on Change

Add a listener mechanism: addChangeListener(ConfigChangeListener l) and a @FunctionalInterface ConfigChangeListener with method onConfigChanged(String key, Object newValue). Whenever a setter is called, notify all registered listeners. AudioSystem should register to listen for volume changes and update its output level automatically.

Why This Matters

This combines Singleton (one config) with Observer (reactive updates). This is how real game engines propagate setting changes — the config is the single source of truth and broadcasts changes to interested systems.

LEVEL 3Challenge 6

Make It Testable — Dependency Injection Refactor

The current design makes unit testing impossible — you can't swap the singleton for a mock. Refactor so systems like AudioSystem accept a GameConfig via constructor. The singleton still exists, but it's injected at the wiring point, not fetched inside classes. Write a unit test that passes a test double.

The Key Lesson

This is the canonical critique of Singleton. The fix is not to abolish it but to inject it instead of having callers fetch it. The object is still created once — it's just no longer fetched with a static call from deep inside your logic.

LEVEL 3Challenge 7

Break and Restore the Singleton via Reflection

Use reflection to call the private constructor and create a second GameConfig instance. Prove that the Singleton guarantee is violated. Then add a guard in the constructor that throws IllegalStateException if an instance already exists, making it reflection-proof. Confirm the guard works.

Think About

This is a real attack vector. Libraries like Mockito use reflection to create objects. Understanding the vulnerability helps you decide when to add the guard — and when it's overkill.

BOSSChallenge 8

Enum Singleton — The Serialisation-Safe Approach

Java's enum guarantees exactly one instance per enum constant — even across serialisation and deserialisation, and even against reflection attacks. Reimplement GameConfig as an enum singleton. Then prove that serialising and deserialising it returns the same instance (not a new copy).

Why This Matters

Joshua Bloch (Effective Java) calls the Enum Singleton "the best way to implement a singleton in Java". Regular Singleton implementations can be broken by serialisation — a deserialized object is a brand new instance with a different identity. Enum prevents this at the language level.

private static final Set<String> VALID_DIFFICULTIES =
    Set.of("EASY", "NORMAL", "HARD");

LEVEL 2Solution 3 — Thread-Safety Stress Test

100 threads, one instance, zero races.

SingletonStressTest.javaJava
import java.util.*;
import java.util.concurrent.*;

public class SingletonStressTest {

    public static void main(String[] args) throws InterruptedException {
        int THREADS = 100;
        Set<GameConfig> seen = Collections.synchronizedSet(new HashSet<>());
        ExecutorService pool = Executors.newFixedThreadPool(THREADS);
        CountDownLatch latch = new CountDownLatch(1); // start all at once

        for (int i = 0; i < THREADS; i++) {
            pool.submit(() -> {
                try { latch.await(); } catch (InterruptedException e) {}
                seen.add(GameConfig.getInstance()); // all threads race here
            });
        }

        latch.countDown(); // release all threads simultaneously
        pool.shutdown();
        pool.awaitTermination(5, TimeUnit.SECONDS);

        System.out.println("Unique instances: " + seen.size());
        // Static Holder → always prints: Unique instances: 1 ✓
        // Naive variant  → may print:   Unique instances: 2+ ✗

        assert seen.size() == 1 : "Singleton violated!";
    }
}
LEVEL 2Solution 4 — Save and Load Config

Properties file persistence on the singleton.

GameConfig.java — persistence methodsJava
public void saveToFile(String path) {
    Properties props = new Properties();
    props.setProperty("volume",     String.valueOf(volume));
    props.setProperty("brightness", String.valueOf(brightness));
    props.setProperty("difficulty", difficulty);
    props.setProperty("resolution", resolution);
    try (FileWriter fw = new FileWriter(path)) {
        props.store(fw, "Game Config");
    } catch (IOException e) {
        System.err.println("Save failed: " + e.getMessage());
    }
}

public void loadFromFile(String path) {
    File f = new File(path);
    if (!f.exists()) {
        System.out.println("No config file found — using defaults.");
        return;
    }
    Properties props = new Properties();
    try (FileReader fr = new FileReader(f)) {
        props.load(fr);
        volume     = Integer.parseInt(props.getProperty("volume",     "70"));
        brightness = Integer.parseInt(props.getProperty("brightness", "80"));
        difficulty = props.getProperty("difficulty", "NORMAL");
        resolution = props.getProperty("resolution", "1920x1080");
    } catch (IOException e) {
        System.err.println("Load failed: " + e.getMessage());
    }
}

// Usage at game startup:
// GameConfig.getInstance().loadFromFile("config.properties");
LEVEL 2Solution 5 — Observer Integration

Singleton as the broadcast hub.

ConfigChangeListener.java + GameConfig.java (updated)Java
@FunctionalInterface
public interface ConfigChangeListener {
    void onConfigChanged(String key, Object newValue);
}

// Inside GameConfig:
private final List<ConfigChangeListener> listeners = new ArrayList<>();

public void addChangeListener(ConfigChangeListener l) { listeners.add(l); }

private void notify(String key, Object val) {
    listeners.forEach(l -> l.onConfigChanged(key, val));
}

public void setVolume(int v) {
    volume = v;
    notify("volume", v); // broadcast to all listeners
}

// AudioSystem registers at startup:
GameConfig.getInstance().addChangeListener((key, val) -> {
    if ("volume".equals(key)) {
        audioEngine.setMasterVolume((Integer) val);
        System.out.println("[Audio] volume updated to: " + val);
    }
});

// Now: GameConfig.getInstance().setVolume(30);
// → [Audio] volume updated to: 30  (automatic, no polling) ✓
LEVEL 3Solution 6 — Make It Testable

Inject the singleton instead of fetching it inside classes.

AudioSystem.java — before and afterJava
// ❌ BEFORE — untestable, hidden dependency
public class AudioSystem {
    public void play() {
        int vol = GameConfig.getInstance().getVolume(); // static call inside logic
        // Can't swap GameConfig in a unit test
    }

// ✓ AFTER — testable, injected dependency
public class AudioSystem {
    private final GameConfig config; // injected, not fetched

    public AudioSystem(GameConfig config) {
        this.config = config;
    }

    public void play() {
        int vol = config.getVolume();
        // In tests: pass a test double. In prod: pass the singleton.
    }
}

// Production wiring:
new AudioSystem(GameConfig.getInstance()); // singleton used ONCE at wiring point

// Unit test (no singleton pollution):
class MockConfig extends GameConfig { /* override getVolume() */ }
new AudioSystem(new MockConfig()); // isolated, reproducible test ✓
The Key Lesson

The singleton still exists. The change is where you call getInstance(). Call it once at the composition root (your main() or DI container), then pass it everywhere as a normal object. Never call getInstance() from deep inside business logic.

LEVEL 3Solution 7 — Break and Reflection-Proof

Expose the vulnerability, then close it.

ReflectionAttack.java + GameConfig.java (hardened)Java
// Breaking it with reflection:
import java.lang.reflect.*;

var ctor = GameConfig.class.getDeclaredConstructor();
ctor.setAccessible(true);
GameConfig evil = (GameConfig) ctor.newInstance();
System.out.println(evil == GameConfig.getInstance()); // false — singleton broken!

// Hardening the constructor:
private GameConfig() {
    // Guard: if the Holder's instance already exists, reject construction
    if (Holder.INSTANCE != null) {
        throw new IllegalStateException(
            "GameConfig already instantiated — use getInstance()");
    }
}

// Now the reflection attack throws IllegalStateException. ✓
//
// Note: this guard can't protect the VERY FIRST construction
// (Holder.INSTANCE is null during first init) — so it only
// blocks attempts after the singleton is established.
// For complete reflection protection → use Enum Singleton (Challenge 8).
BOSSSolution 8 — Enum Singleton

The safest Java Singleton. Serialisation-proof and reflection-proof by the language itself.

GameConfig.java — Enum SingletonJava
public enum GameConfig {

    INSTANCE; // the single instance — declared as an enum constant

    private int    volume     = 70;
    private int    brightness = 80;
    private String difficulty = "NORMAL";
    private String resolution = "1920x1080";

    public int    getVolume()            { return volume; }
    public void   setVolume(int v)        { volume = v; }
    public int    getBrightness()         { return brightness; }
    public void   setBrightness(int b)     { brightness = b; }
    public String getDifficulty()          { return difficulty; }
    public void   setDifficulty(String d)  { difficulty = d; }
    public String getResolution()           { return resolution; }
    public void   setResolution(String r)  { resolution = r; }

    @Override
    public String toString() {
        return String.format("[CONFIG] vol:%d bright:%d diff:%s res:%s",
            volume, brightness, difficulty, resolution);
    }
}

// Usage — identical to before, just different syntax:
// GameConfig.INSTANCE.setVolume(50);
// GameConfig.INSTANCE.getVolume();  → 50

// Proving serialisation-safety:
// Serialise:
var bos = new ByteArrayOutputStream();
new ObjectOutputStream(bos).writeObject(GameConfig.INSTANCE);

// Deserialise:
var restored = (GameConfig) new ObjectInputStream(
    new ByteArrayInputStream(bos.toByteArray())).readObject();

System.out.println(restored == GameConfig.INSTANCE); // true ✓
// A class-based Singleton would print false here — enum prevents
// deserialisation from creating a new instance. Language-level guarantee.
Real-World Connections

Java Runtime: Runtime.getRuntime() is a classic Singleton — one JVM runtime, globally accessible.

Spring Framework: By default, all Spring beans are singletons within the ApplicationContext. The container manages the single instance and injects it where needed — this is Singleton + Dependency Injection combined.

Android: Application.getApplicationContext() follows the Singleton pattern — one app context for the entire app lifecycle.

Logging: Logger.getLogger("name") returns a cached logger — effectively a registry of singletons, one per name.
