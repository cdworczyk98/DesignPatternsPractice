// Step 02

New Requirements That Break It

The rental company expands. New vehicle types arrive. The creation logic is duplicated everywhere — and every new type requires hunting through the whole codebase.

New Requirements

Add Van, ElectricCar, and Scooter. The if/else creation block now exists in 4 different files. Adding "Van" means editing all four.

Creation logic duplicated everywhereJava
// BookingService.java
if      (type.equals("car"))        v = new Car();
else if (type.equals("truck"))      v = new Truck();
else if (type.equals("motorcycle")) v = new Motorcycle();
// add Van here...

// ReservationAPI.java — same block, copy-pasted
// AdminPanel.java     — same block again
// TestSuite.java      — same block again
// → forget one? Wrong vehicle created silently.
Violation

Open/Closed Principle

Adding Van requires modifying every file that has a creation block. Classes should be open for extension, closed for modification.

Violation

DRY Principle

Same if/else creation logic copy-pasted into multiple classes. One missed update is a bug that is nearly impossible to trace.

Violation

Single Responsibility

BookingService should book vehicles, not decide how to construct them. Construction knowledge is not its job.

Violation

Dependency Inversion

High-level modules directly depend on concrete classes (new Car(), new Truck()). They should depend on abstractions.

The Core Insight

The Factory Pattern's job is simple: centralise object creation. Instead of callers knowing how to build a Vehicle, they ask a Factory for a Vehicle. The caller never touches new Car() directly.

// Step 03

Deep Dive — Three Factory Variants

"Factory" is actually a family of three related patterns. Understanding when to use each one is the real skill.

Static Factory
aka "Simple Factory"

A single class with a static method. Not a GoF pattern — just a useful helper. Great starting point.

Simplest
Factory Method
GoF Creational Pattern

An abstract creator class with a createVehicle() method. Subclasses decide what to create.

Most Common
Abstract Factory
GoF Creational Pattern

A factory of factories. Creates families of related objects that must be compatible with each other.

Most Powerful

Diagram 1 — Static Factory

The client calls one static method. All new keywords live inside that one method. Nothing more.

Diagram 2 — Factory Method Pattern

An abstract creator defines createVehicle() as abstract. Concrete subclasses implement it. The parent holds the workflow; children decide what to build.

Diagram 3 — Abstract Factory Pattern

Two parallel hierarchies: factories and products. Concrete factories (ElectricFleet, CombustionFleet) produce entire compatible families of vehicles.

// When to Use Which
  1. Use Static FactoryCreation logic is simple and stable. Good for utilities and getting started. Not GoF — just a clean helper.
  2. Use Factory MethodDifferent contexts need different implementations. A city agency creates Cars; a freight agency creates Trucks. The pattern delegates the "which" to subclasses while keeping the workflow in the parent.
  3. Use Abstract FactoryYou need entire families of related objects that must stay consistent. An electric fleet should never accidentally mix with a combustion vehicle. Abstract Factory enforces family-level consistency.
  4. Common mistake: over-engineeringStart with Static Factory. Graduate to Factory Method when you need polymorphic construction. Reach for Abstract Factory only when you truly have multiple product families. Never jump straight to Abstract Factory for a three-vehicle system.
The Dependency Inversion Principle

High-level modules should not depend on low-level modules. Before the factory, BookingService calls new Car() — coupled to concrete. After, it calls factory.create("car") — depends only on Vehicle. That is Dependency Inversion in action.

Refactored — All Three Variants

All three factory variants implemented on the same domain so you can compare them side by side.

Variant A — Static Factory

One class, one static method. All new keywords in one place.

VehicleFactory.java — Static FactoryJava
public class VehicleFactory {

    private VehicleFactory() {} // prevent instantiation

    public static Vehicle create(String type) {
        return switch (type.toLowerCase()) {
            case "car"        -> new Car();
            case "truck"      -> new Truck();
            case "motorcycle" -> new Motorcycle();
            case "van"        -> new Van();         // add new types HERE
            case "electric"   -> new ElectricCar(); // only HERE
            default          ->
                throw new IllegalArgumentException("Unknown: " + type);
        };
    }
}

// BookingService, ReservationAPI, AdminPanel — all now call:
// Vehicle v = VehicleFactory.create(type);
// Not a single 'new Car()' outside this class.
Variant B — Factory Method Pattern

Abstract creator class. The parent defines the booking workflow; subclasses decide what vehicle to build.

RentalAgency.java — Abstract CreatorJava
public abstract class RentalAgency {

    // THE factory method — subclasses MUST implement this
    protected abstract Vehicle createVehicle();

    // Template method — uses the factory method internally
    public String bookVehicle(String customer) {
        Vehicle v = createVehicle(); // delegates to subclass
        return String.format("[BOOKING] %s reserved a %s at %s",
            customer, v.getType(), getClass().getSimpleName());
    }
}

}

public class FreightAgency extends RentalAgency {
    @Override protected Vehicle createVehicle() { return new Truck(); }
}

public class SpeedAgency   extends RentalAgency {

Variant C — Abstract Factory Pattern

Creates families of compatible vehicles. Swapping one factory swaps the entire fleet.

public interface FleetFactory {
    Vehicle createCityVehicle();

public class ElectricFleet implements FleetFactory {

RookieChallenge 1

RookieChallenge 2

Registry-Based Factory

Refactor VehicleFactory to use a Map<String, Supplier<Vehicle>> registry. New vehicle types can be registered with VehicleFactory.register("van", Van::new) — without ever modifying the factory class again.

Think About

This makes the factory fully Open/Closed. What are the trade-offs vs a switch statement? When would you prefer each approach?

ProChallenge 3

Parameterised Factory Method

Extend the Factory Method variant so createVehicle() accepts a VehicleConfig record with colour, plateNumber, and isAutomatic. Update CityAgency and FreightAgency to pass config down to their concrete vehicles.

Think About

What is the cleanest way to pass config to each vehicle class? Should the vehicle store the config or extract individual fields?

ProChallenge 4

Logging Factory Wrapper

Create a LoggingVehicleFactory that wraps VehicleFactory. Every call to create() logs the type requested, a timestamp, and the class name of the created object. The rest of the codebase should not need to change.

Hint

This combines Factory with Proxy. Extract a IVehicleFactory interface so both implementations are swappable to the caller.

ProChallenge 5

Add HybridFleet to the Abstract Factory

Add a HybridFleet that creates HybridCar (city), HybridTruck (highway), HybridVan (delivery). Verify that swapping new ElectricFleet() with new HybridFleet() is the only client change required.

Goal

This is the key demonstration of Abstract Factory power: swap an entire product family with one line.

EliteChallenge 6

Config File Factory (Reflection)

Load vehicle type mappings from a vehicles.properties file (car=com.rental.Car, etc.). Use Class.forName() to instantiate. Adding "hovercraft" to the app requires only a new class and a new properties line — zero Java code changes.

Think About

This is how Spring's BeanFactory works at its core. What are the risks of reflection-based factories? (Type safety, error messages, performance.)

EliteChallenge 7

Factory Method + Template Method Combo

Extend RentalAgency so bookVehicle() is a full Template Method with steps: validateCustomer() → createVehicle() → prepareVehicle() → issueReceipt(). Make the first two overridable with defaults. FreightAgency should override validateCustomer() to require a commercial licence prefix.

Why This Matters

Factory Method is almost always found inside a Template Method in real codebases. This is the backbone of Spring's AbstractApplicationContext and JUnit's test lifecycle.

BossChallenge 8

New Domain — UI Component Abstract Factory

Redesign Abstract Factory for a cross-platform UI toolkit. UIFactory creates: Button, TextField, Dropdown. Implement WindowsUIFactory and MacUIFactory. The app's rendering code uses only UIFactory — swapping platform is one line.

Why This Matters

This is the canonical GoF Abstract Factory example. It is also exactly how Java Swing's pluggable look-and-feel system works. Finishing this means you fully understand the pattern.

RookieSolution 2 — Registry-Based Factory

Map<String, Supplier<Vehicle>> makes the factory truly Open/Closed.

VehicleFactory.java — Registry approachJava
import java.util.*;
import java.util.function.*;

public class VehicleFactory {
    private static final Map<String, Supplier<Vehicle>> REGISTRY = new HashMap<>();

    static {
        REGISTRY.put("car",        Car::new);
        REGISTRY.put("truck",      Truck::new);
        REGISTRY.put("motorcycle", Motorcycle::new);
    }

    // External code can register new types — factory never changes again
    public static void register(String type, Supplier<Vehicle> s) {
        REGISTRY.put(type.toLowerCase(), s);
    }

    public static Vehicle create(String type) {
        Supplier<Vehicle> s = REGISTRY.get(type.toLowerCase());
        if (s == null) throw new IllegalArgumentException("Unknown: " + type);
        return s.get();
    }
}

// Anywhere in startup — no factory edit required:
// VehicleFactory.register("hovercraft", Hovercraft::new);
// Vehicle h = VehicleFactory.create("hovercraft"); ✓
Switch vs Registry

Switch: compile-time safe, easy to read, fine for stable types. Registry: fully Open/Closed, supports plugins. Trade-off: type errors surface at runtime not compile time. Prefer switch for internal types, registry for plugin systems.

// Java 16+ record — immutable config carrier
public record VehicleConfig(String colour, String plateNumber, boolean isAutomatic) {}

public abstract class RentalAgency {
    protected abstract Vehicle createVehicle(VehicleConfig config);

    public String bookVehicle(String customer, VehicleConfig config) {
        Vehicle v = createVehicle(config);
        return String.format("[%s] %s booked %s plate:%s colour:%s auto:%b",
            getClass().getSimpleName(), customer,
            v.getType(), config.plateNumber(), config.colour(), config.isAutomatic());
    }
}

public class CityAgency extends RentalAgency {
    @Override
    protected Vehicle createVehicle(VehicleConfig config) {
        return new Car(config); // Car constructor accepts config
    }
}
ProSolution 4 — Logging Factory Wrapper

Proxy wraps the factory transparently via a shared interface.

IVehicleFactory.java + LoggingVehicleFactory.javaJava
public interface IVehicleFactory { Vehicle create(String type); }

public class VehicleFactory implements IVehicleFactory { /* existing logic */ }

public class LoggingVehicleFactory implements IVehicleFactory {
    private final IVehicleFactory delegate;

    public LoggingVehicleFactory(IVehicleFactory delegate) {
        this.delegate = delegate;
    }

    @Override
    public Vehicle create(String type) {
        System.out.printf("[%d] Requesting: %s%n", System.currentTimeMillis(), type);
        Vehicle v = delegate.create(type);
        System.out.printf("[%d] Created: %s%n", System.currentTimeMillis(), v.getClass().getSimpleName());
        return v;
    }
}

// Only wiring point changes — call sites stay identical:
// IVehicleFactory factory = new LoggingVehicleFactory(new VehicleFactory());
// factory.create("car"); // logs transparently ✓

EliteSolution 6 — Config File + Reflection Factory

Zero Java changes needed to add new vehicle types.

ReflectiveVehicleFactory.javaJava
import java.io.*;
import java.util.*;

public class ReflectiveVehicleFactory {
    private final Properties props = new Properties();

    public ReflectiveVehicleFactory(String file) throws IOException {
        try (InputStream in = getClass().getClassLoader().getResourceAsStream(file)) {
            props.load(in);
        }
    }

    public Vehicle create(String type) {
        String className = props.getProperty(type.toLowerCase());
        if (className == null) throw new IllegalArgumentException("No mapping for: " + type);
        try {
            return (Vehicle) Class.forName(className).getDeclaredConstructor().newInstance();
        } catch (Exception e) {
            throw new RuntimeException("Failed to create: " + className, e);
        }
    }
}
// vehicles.properties:
// car=com.rental.Car
// truck=com.rental.Truck
// hovercraft=com.rental.Hovercraft   ← just add this line, no Java changes
//
// Risk: ClassCastException at runtime if class doesn't implement Vehicle.
// This is how Spring's BeanFactory resolves beans from XML config. ✓
EliteSolution 7 — Factory Method + Template Method

Template defines the invariant steps; subclasses override hooks.

RentalAgency.java — full Template MethodJava
public abstract class RentalAgency {

    // Final template — the invariant algorithm
    public final String bookVehicle(String customer) {
        validateCustomer(customer);      // hook
        Vehicle v = createVehicle();      // factory method
        prepareVehicle(v);               // hook
        return issueReceipt(customer, v); // final step
    }

    protected abstract Vehicle createVehicle(); // must implement

    protected void validateCustomer(String c) {
        if (c == null || c.isBlank())
            throw new IllegalArgumentException("Name required");
    }
    protected void prepareVehicle(Vehicle v) { /* default: no-op */ }

    private String issueReceipt(String c, Vehicle v) {
        return "Receipt: " + c + " booked " + v.getType();
    }
}

// FreightAgency overrides two hooks:
public class FreightAgency extends RentalAgency {
    @Override protected Vehicle createVehicle() { return new Truck(); }

    @Override
    protected void validateCustomer(String c) {
        super.validateCustomer(c);
        if (!c.startsWith("COMM-"))
            throw new SecurityException("Commercial licence required");
    }

    @Override
    protected void prepareVehicle(Vehicle v) {
        System.out.println("[Freight] Safety inspection: " + v.getType());
    }
}
BossSolution 8 — UI Component Abstract Factory

The canonical GoF example. Same structure, new domain.

UIFactory.java + all components + concrete factoriesJava
// Product interfaces
public interface Button    { String render(); }
public interface TextField { String render(); }
public interface Dropdown  { String render(); }

// Abstract Factory
public interface UIFactory {
    Button    createButton();
    TextField createTextField();
    Dropdown  createDropdown();
}

// Windows products
class WinButton    implements Button    { public String render() { return "[Win Button]"; } }
class WinTextField implements TextField { public String render() { return "[Win Input ]"; } }
class WinDropdown  implements Dropdown  { public String render() { return "[Win Combo ]"; } }

// Mac products
class MacButton    implements Button    { public String render() { return "(Mac Button)"; } }
class MacTextField implements TextField { public String render() { return "(Mac Input )"; } }
class MacDropdown  implements Dropdown  { public String render() { return "(Mac Select)"; } }

// Concrete factories
public class WindowsUIFactory implements UIFactory {
    public Button    createButton()    { return new WinButton(); }
    public TextField createTextField() { return new WinTextField(); }
    public Dropdown  createDropdown()  { return new WinDropdown(); }
}

public class MacUIFactory implements UIFactory {
    public Button    createButton()    { return new MacButton(); }
    public TextField createTextField() { return new MacTextField(); }
    public Dropdown  createDropdown()  { return new MacDropdown(); }
}

// Application — never touches concrete classes:
public class Application {
    private final UIFactory ui;
    public Application(UIFactory ui) { this.ui = ui; }
    public void renderForm() {
        System.out.println(ui.createButton().render());
        System.out.println(ui.createTextField().render());
        System.out.println(ui.createDropdown().render());
    }
}

// new Application(new WindowsUIFactory()).renderForm();
// → [Win Button] [Win Input ] [Win Combo ]
//
// new Application(new MacUIFactory()).renderForm();
// → (Mac Button) (Mac Input ) (Mac Select)
//
// This is exactly how Java Swing's UIManager.setLookAndFeel() works. ✓
Real-World Connections

Java Swing: UIManager.setLookAndFeel() swaps the entire Abstract Factory that creates platform-native components.

Spring Framework: ApplicationContext is an Abstract Factory — it creates beans according to XML/annotation configuration (the concrete factory).

JDBC: DriverManager.getConnection() is a Factory Method. Connection.createStatement() is also a Factory Method — Connection is an abstract creator producing Statement objects.
