public abstract class Dish {
    protected String       name;
    protected List<String> ingredients;
    protected double       price;

    public abstract void prepare();
    public abstract void plate();

    public String getName()          { return name; }
    public double getPrice()         { return price; }
    public List<String> getIngredients() { return ingredients; }

    public void printReceipt() {
        System.out.printf("  %-28s £%.2f%n", name, price);
    }
}

OrderProcessor.java — the creation problem lives hereJava
public class OrderProcessor {

    public Dish processOrder(String cuisine, String dishName) {
        Dish dish;

        // Creation logic directly in the processor — this is the problem
        if (cuisine.equals("italian")) {
            switch (dishName) {
                case "pizza":      dish = new MargheritaPizza();    break;
                case "pasta":      dish = new TagliatelleRagu();     break;
                case "dessert":    dish = new Tiramisu();            break;
                default: throw new IllegalArgumentException("Unknown dish");
            }
        } else if (cuisine.equals("japanese")) {
            switch (dishName) {
                case "ramen":     dish = new TonkotsuRamen();      break;
                case "sashimi":   dish = new SalmonSashimi();     break;
                case "dessert":   dish = new MochiIceCream();     break;
                default: throw new IllegalArgumentException("Unknown dish");
            }
        } else if (cuisine.equals("vegan")) {
            switch (dishName) {
                case "bowl":    dish = new BuddhaBowl();        break;
                case "burger":  dish = new JackfruitBurger();   break;
                case "dessert": dish = new ChiaPudding();      break;
                default: throw new IllegalArgumentException("Unknown dish");
            }
        } else {
            throw new IllegalArgumentException("Unknown cuisine: " + cuisine);
        }

        dish.prepare();
        dish.plate();
        return dish;
    }
}

New Requirements That Break It

The restaurant group expands. A Mexican kitchen opens next door. A premium Tasting Menu is added with custom creation logic per dish. A third-party delivery app needs to order dishes by ID string from a config file. None of these fit the current design without major surgery.

New Requirements

① Mexican kitchen — add Tacos, Guacamole, Churros. ② Tasting Menu — dishes need custom pairing logic and wine suggestions (different creation behaviour per restaurant). ③ Config-driven ordering — a delivery app sends "italian:pizza" as a string; the system must resolve it without a giant switch. ④ Test isolation — unit tests for OrderProcessor must not instantiate real dishes (too expensive, hits a real DB).

Violation

Open/Closed Principle

Adding any new cuisine, new dish, or new restaurant branch requires editing OrderProcessor. A class that grows every time the menu expands is not closed for modification.

Violation

Single Responsibility

OrderProcessor knows how to process orders and knows which concrete class maps to every cuisine/dish combination. Two jobs, one class, guaranteed coupling.

Violation

No Branch Variation

The Shoreditch branch makes dishes differently from the City branch. The current design has no way to vary creation logic per location without duplicating the entire processor.

Violation

Untestable in Isolation

To unit-test order processing logic, you must instantiate real dishes. You can't inject mock dishes without restructuring the entire creation chain.

The Core Insight

Creation logic and usage logic are two separate concerns. Separate the "what to build" from the "how to use it". A factory knows how to construct objects. The processor just asks the factory for a dish — it never calls new directly. New cuisines mean new factories, not edits to the processor.

// step_03

Deep Dive

The Factory Pattern has three distinct forms. Knowing when to use each — and understanding what problem each solves — is the real skill. All three appear in production Java code daily.

Diagram 1 — Three Factory Forms

Think of the three forms as a spectrum from simplest to most powerful. Static Factory centralises creation. Factory Method lets subclasses decide. Abstract Factory ensures families of objects are compatible.

Diagram 2 — Factory Method Structure

The Factory Method pattern defines an interface for creating an object but lets subclasses decide which class to instantiate. The abstract creator class contains the template method (orderDish) that calls the abstract factory method (createDish). Subclasses implement only the factory method.

Diagram 3 — Abstract Factory: Ensuring Compatible Families

Abstract Factory ensures that a set of related objects are created consistently. A Japanese kitchen's starter, main, and dessert all belong together — you can't mix a sashimi starter with a tiramisu dessert. The factory enforces this constraint at compile time.

// design_principles_at_play
  1. Dependency Inversion PrincipleOrderProcessor used to depend on MargheritaPizza, TonkotsuRamen — concrete classes. After the refactor it depends on Dish and a factory interface — abstractions. Both high-level and low-level modules depend on abstractions, not each other.
  2. Open/Closed PrincipleAdding a Mexican kitchen means creating MexicanRestaurant extends Restaurant and three new dish classes. Zero edits to OrderProcessor, zero edits to existing restaurants. The system is open for extension, closed for modification.
  3. Single Responsibility PrincipleCreation is separated from usage. OrderProcessor processes orders. Factories build dishes. Restaurant subclasses know their own menu. Each class has one reason to change.
  4. Template Method (bonus pattern)The Factory Method pattern is almost always paired with Template Method. Restaurant.orderDish() is the template — it defines the algorithm (prepare, plate, serve). createDish() is the hook that subclasses fill in. You get two patterns for the price of one.
  5. Registry Pattern (static factory variant)Instead of a switch statement, store Map<String, Supplier<Dish>>. Register entries at startup. Adding a new dish is one line: registry.put("vegan:wrap", VeggieWrap::new). No switch, no recompilation of the factory class.
The Restaurant Industry Parallel

A real restaurant franchise is a perfect Factory Method analogy. Corporate HQ defines the ordering process: take order → prepare → plate → serve. Each branch overrides only the preparation step — the Shoreditch branch uses artisan ingredients, the motorway services branch uses pre-made components. Same process, different factories, complete independence.

// form_1 — static factory
DishFactory.java — Static Factory with RegistryJava
import java.util.*;
import java.util.function.*;

public class DishFactory {

    // Registry: "cuisine:dish" → Supplier (lambda, no args constructor reference)
    // Adding a new dish is one line — no switch, no recompilation of this class
    private static final Map<String, Supplier<Dish>> REGISTRY = new HashMap<>();

    static {
        // Italian
        REGISTRY.put("italian:pizza",   MargheritaPizza::new);
        REGISTRY.put("italian:pasta",   TagliatelleRagu::new);
        REGISTRY.put("italian:dessert", Tiramisu::new);
        // Japanese
        REGISTRY.put("japanese:ramen",   TonkotsuRamen::new);
        REGISTRY.put("japanese:sashimi", SalmonSashimi::new);
        REGISTRY.put("japanese:dessert", MochiIceCream::new);
        // Vegan
        REGISTRY.put("vegan:bowl",    BuddhaBowl::new);
        REGISTRY.put("vegan:burger",  JackfruitBurger::new);
        REGISTRY.put("vegan:dessert", ChiaPudding::new);
    }

    // Allow runtime registration — delivery app can add dishes from config
    public static void register(String key, Supplier<Dish> supplier) {
        REGISTRY.put(key, supplier);
    }

    public static Dish create(String cuisine, String dishName) {
        String key = cuisine.toLowerCase() + ":" + dishName.toLowerCase();
        Supplier<Dish> supplier = REGISTRY.get(key);
        if (supplier == null)
            throw new IllegalArgumentException("Unknown dish: " + key);
        return supplier.get(); // creates fresh instance via Supplier
    }

    public static Set<String> availableDishes() { return REGISTRY.keySet(); }
}

Restaurant.java — Abstract CreatorJava
/**
 * The Creator. Contains the template method (orderDish) and
 * declares the factory method (createDish) as abstract.
 * Subclasses only override createDish — the order workflow is inherited.
 */
public abstract class Restaurant {

    // Template method — the algorithm stays here, fixed
    public Dish orderDish(String dishName) {
        System.out.println("\n=== Order: " + dishName + " at " + getName() + " ===");
        Dish dish = createDish(dishName); // ← calls subclass implementation
        dish.prepare();
        dish.plate();
        dish.printReceipt();
        return dish;
    }

    // Factory method — subclasses decide what to instantiate
    protected abstract Dish createDish(String dishName);

    public abstract String getName();
}

KitchenFactory.java — Abstract FactoryJava
// Product interfaces — one per course type
public interface Starter    { void serve(); }
public interface MainCourse { void serve(); }
public interface Dessert    { void serve(); }

// Abstract Factory — defines the "family" contract
public interface KitchenFactory {
    Starter    createStarter();
    MainCourse createMain();
    Dessert    createDessert();
}

// Italian family — all three products belong together
public class ItalianKitchen implements KitchenFactory {
    @Override public Starter    createStarter()  { return new Bruschetta(); }
    @Override public MainCourse createMain()     { return new MargheritaPizza(); }
    @Override public Dessert    createDessert()  { return new Tiramisu(); }
}

// Japanese family — guaranteed consistent
public class JapaneseKitchen implements KitchenFactory {
    @Override public Starter    createStarter()  { return new MisoSoup(); }
    @Override public MainCourse createMain()     { return new TonkotsuRamen(); }
    @Override public Dessert    createDessert()  { return new MochiIceCream(); }
}

// Tasting menu orchestrator — only knows KitchenFactory, never concrete classes
public class TastingMenu {
    private final KitchenFactory kitchen;

    public TastingMenu(KitchenFactory kitchen) { this.kitchen = kitchen; }

    public void serve() {
        kitchen.createStarter().serve();  // always the right family
        kitchen.createMain().serve();
        kitchen.createDessert().serve();
    }
}
// new TastingMenu(new ItalianKitchen()).serve();  → Italian family ✓
// new TastingMenu(new JapaneseKitchen()).serve(); → Japanese family ✓
// Can't accidentally mix families — compiler prevents it ✓

Allergen-Aware Factory

Think About

The allergen check happens in the factory, before the object is constructed. This is a key advantage of factories over direct new calls — you can add pre-construction validation in one place that applies to every caller.

CommisChallenge 3

Config-File–Driven Registration

Load dish registrations from a dishes.properties file at startup: italian:pizza=MargheritaPizza. Use Class.forName() and reflection to instantiate the class from the string. Why This Matters

This is how Spring's ApplicationContext works at its core. Bean definitions in XML/YAML are properties files. Spring reads them, uses reflection to instantiate the class, and registers the bean. You are building a micro Spring.

Sous ChefChallenge 4

Premium Branch — Factory Method Variation

Create a PremiumItalianRestaurant extends ItalianRestaurant. Sous ChefChallenge 5

Parametrised Factory — Custom Pasta

Some dishes need parameters at creation time. Create a CustomPasta class that takes a sauce type (enum Sauce { BOLOGNESE, CARBONARA, PESTO, ARRABIATA }) and a pasta shape (enum Shape { TAGLIATELLE, SPAGHETTI, RIGATONI }) in the constructor. The standard factory method takes no parameters (or just a name). Parameterised creation pushes you toward a Builder pattern or a factory that accepts a config record. How you bridge them is the design challenge.

Sous ChefChallenge 6

Cached / Pooled Factory

Some dishes are stateless (menus, ingredient lists). Repeatedly creating new instances wastes memory. Add a CachedDishFactory that wraps DishFactory: on the first request for a given key it creates the dish and caches it; subsequent requests return the same instance. Make it thread-safe. When would caching dishes be correct and when would it be wrong?

Think About

Caching is only safe for immutable or stateless objects. A dish whose prepare() modifies internal state must not be shared. This is the same trade-off as the Flyweight pattern and connection pooling.

Sous ChefChallenge 7

Factory + Decorator — Dietary Modifiers

Use the Decorator pattern on top of the Factory. Create a DishDecorator abstract class extends Dish. Then: GlutenFreeDecorator (removes gluten ingredients, adds £1.50 surcharge), SpicyDecorator (adds chilli to ingredient list). Chain them: new SpicyDecorator(new GlutenFreeDecorator(DishFactory.create("italian","pizza"))). The factory creates the base; decorators modify it.

Why This Matters

Factory + Decorator is one of the most common pattern combinations in real code. Spring's bean post-processors wrap beans with decorators after factory creation. This is also how Java I/O streams work.

Head ChefChallenge 9

Test with a Mock Factory

The whole point of factories is testability. Write a MockRestaurant extends Restaurant whose createDish() returns a MockDish — a simple stub that tracks whether prepare() and plate() were called. BOSSChallenge 10

New Domain — JDBC Connection Factory

Port all three factory forms to database connections. Static Factory: ConnectionFactory.create("mysql") → MySqlConnection. Factory Method: abstract DataAccessLayer with createConnection() — subclasses provide PostgreSQL vs MySQL connections, the query template stays in the parent. public class AllergenException extends RuntimeException {
    public AllergenException(String dish, String allergen) {
        super("Dish '" + dish + "' contains allergen: " + allergen);
    }

// In DishFactory — safe creation method
public static Dish createSafe(String cuisine, String dishName,
                               Set<String> avoidAllergens) {
    Dish dish = create(cuisine, dishName); // create first to inspect allergens
    for (String allergen : avoidAllergens) {
        if (dish.getAllergens().contains(allergen)) {
            throw new AllergenException(dishName, allergen);
        }
    }
    return dish;
}

CommisSolution 3 — Config-File Registration

Reflection-based factory registration. The skeleton of Spring.

DishFactoryLoader.javaJava
// dishes.properties file content:
// italian:pizza=com.restaurant.dishes.MargheritaPizza
// italian:pasta=com.restaurant.dishes.TagliatelleRagu
// japanese:ramen=com.restaurant.dishes.TonkotsuRamen
// ...

public class DishFactoryLoader {

    public static void loadFromProperties(String resourcePath) {
        Properties props = new Properties();

        try (var in = DishFactoryLoader.class
                .getResourceAsStream(resourcePath)) {
            props.load(in);
        } catch (IOException e) {
            throw new RuntimeException("Cannot load dish config", e);
        }

        props.forEach((key, value) -> {
            final String className = (String) value;
            // Register a Supplier that uses reflection to instantiate
            DishFactory.register((String) key, () -> {
                try {
                    return (Dish) Class.forName(className)
                                        .getDeclaredConstructor()
                                        .newInstance();
                } catch (Exception e) {
                    throw new RuntimeException("Cannot instantiate: " + className, e);
                }
            });
        });
    }
}

// Startup:
// DishFactoryLoader.loadFromProperties("/dishes.properties");
// Dish d = DishFactory.create("italian", "pizza"); // works ✓
//
// To add a new dish: edit dishes.properties, no recompile. ✓
//
// This is Spring's ClassPathXmlApplicationContext at its core:
//   - Read bean definitions from XML/YAML/properties
//   - Use Class.forName() + newInstance() to create beans
//   - Register in ApplicationContext (our DishFactory registry)
// Spring adds scope, lifecycle, AOP proxying — but the skeleton is identical.

Sous ChefSolution 6 — Cached Factory

Thread-safe cache — correct only for stateless/immutable dishes.

CachedDishFactory.javaJava
public class CachedDishFactory {

    // ConcurrentHashMap: thread-safe reads with fine-grained locking on writes
    private final Map<String, Dish> cache = new ConcurrentHashMap<>();

    public Dish create(String cuisine, String dishName) {
        String key = cuisine + ":" + dishName;
        // computeIfAbsent is atomic — only one thread creates per key
        return cache.computeIfAbsent(key, k -> DishFactory.create(cuisine, dishName));
    }

    public void evict(String key) { cache.remove(key); }
    public void clearAll()         { cache.clear(); }
    public int  cacheSize()        { return cache.size(); }
}

// When caching is SAFE:
//   - The dish object is immutable after construction (no setters)
//   - prepare() and plate() are pure (no side effects on dish state)
//   - Dish is used as a value object (recipe card, menu entry)
//   Example: DishMetadata, IngredientList, NutritionalInfo
//
// When caching is WRONG:
//   - prepare() sets instance state (e.g. this.cooked = true)
//   - The dish tracks runtime data (temperature, status)
//   - You need a FRESH instance per order (customer A's pizza ≠ customer B's pizza)
//   Example: every real dish ordered — must be a new instance
//
// This is exactly the Flyweight pattern: share immutable state,
// don't share mutable state. Connection pooling uses the same logic:
// pool Connection objects (reusable after reset), never pool active transactions.

You've been using all three Factory forms in every JDBC query.

All three Factory forms mapped to real JDBC APIJava
// ═══ FORM 1: STATIC FACTORY ═══════════════════════════════
// DriverManager.getConnection() is a Static Factory Method.
// You pass a URL string; it resolves the correct Driver and creates a Connection.
Connection conn = DriverManager.getConnection("jdbc:mysql://localhost/db", user, pass);
// ↑ Equivalent to our DishFactory.create("mysql", ...) ✓

// ═══ FORM 2: FACTORY METHOD ════════════════════════════════
// DataSource is an abstract Creator. HikariDataSource is a ConcreteCreator.
// getConnection() is the factory method. The connection pool management
// logic (template) stays in DataSource; HikariDataSource supplies real connections.
DataSource ds = new HikariDataSource(config); // concrete creator
Connection c  = ds.getConnection();             // factory method
// ↑ Equivalent to our restaurant.createDish("pizza") ✓
// Template = pool management logic in DataSource
// Factory method = HikariDataSource.getConnection() creates real pooled connection

// ═══ FORM 3: ABSTRACT FACTORY ══════════════════════════════
// Connection creates Statement and PreparedStatement — a family of compatible objects.
// MySQLConnection creates MySQL-compatible Statement/PreparedStatement.
// PostgreSQLConnection creates Postgres-compatible ones.
// You can't mix a MySQL Statement with a PostgreSQL Connection.
Connection         conn2 = ds.getConnection();        // Abstract Factory
Statement          stmt  = conn2.createStatement();    // createMain()
PreparedStatement  pstmt = conn2.prepareStatement(sql);// createDish(name)
CallableStatement  cstmt = conn2.prepareCall(sql);     // createStarter()
// ↑ Equivalent to our KitchenFactory creating compatible Starter/Main/Dessert ✓
//
// Summary table:
// OUR PATTERN               JDBC EQUIVALENT
// ─────────────────────────────────────────────────────────────
// DishFactory.create()      DriverManager.getConnection(url)
// Restaurant (abstract)     javax.sql.DataSource
// ItalianRestaurant         HikariDataSource / C3P0DataSource
// KitchenFactory            java.sql.Connection

Real-World Connections

Spring Framework: BeanFactory is a Static Factory. ApplicationContext is a Factory Method creator — subclasses like ClassPathXmlApplicationContext and AnnotationConfigApplicationContext override createBeanFactory(). @Configuration classes with @Bean methods are Abstract Factories.

Java standard library: Calendar.getInstance(), NumberFormat.getInstance(), Collections.unmodifiableList() — all static factory methods. Iterator returned by Collection.iterator() — factory method on the collection.

Effective Java (Bloch): Item 1 is "Consider static factory methods instead of constructors." They have names (unlike constructors), can return subtypes, can cache instances, and don't require creating a new object each time. Boolean.valueOf(true) never creates a new object.

Mockito / JUnit: Mockito.mock(MyClass.class) is a Static Factory that creates a proxy subclass at runtime. The entire testing infrastructure is built on Factory + Proxy patterns.
