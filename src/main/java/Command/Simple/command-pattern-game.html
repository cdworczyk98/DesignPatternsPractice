public class Hero {
    private int hp           = 100;
    private int mp           = 60;
    private int attackPower  = 15;
    private int x            = 0;
    private int y            = 0;

    public void attack(Enemy enemy) {
        int dmg = attackPower;
        enemy.takeDamage(dmg);
        System.out.println("Hero attacks for " + dmg);
    }

    public void castSpell(Enemy enemy) {
        if (mp < 20) { System.out.println("Not enough MP!"); return; }
        mp -= 20;
        enemy.takeDamage(30);
        System.out.println("Fireball! 30 damage, MP now: " + mp);
    }

    public void move(int dx, int dy) {
        x += dx; y += dy;
        System.out.println("Hero moves to (" + x + "," + y + ")");
    }

    // getters omitted for brevity
}
Enemy.javaJava
public class Enemy {
    private String name;
    private int    hp      = 80;
    private int    defence = 5;

    public Enemy(String name) { this.name = name; }

    public void takeDamage(int dmg) {
        int actual = Math.max(0, dmg - defence);
        hp -= actual;
        System.out.println(name + " takes " + actual + " dmg. HP: " + hp);
    }

    public void heal(int amount) { hp = Math.min(80, hp + amount); }
    public int  getHp()          { return hp; }
    public boolean isAlive()     { return hp > 0; }
}

Works — For Now

Clean and simple. The limitation is that actions are fire-and-forget. Once a method is called, that action is gone — no record of it, no way to reverse it, no way to delay or replay it. Click ② to see why this becomes painful fast.

// step_02

New Requirements That Break It

The game needs an undo system (misclick recovery), an action replay system (cutscenes), and an input queue (turn-based scheduling). None of these are possible when actions are plain method calls.

New Requirements

① Undo: Player misclicked Cast Spell — give them a second. ② Replay: Record and replay the winning battle as a cinematic. ③ Queue: Schedule actions for a turn-based system — execute them at the right moment. ④ Macro: Record a combo (attack → move → spell) and replay with one button.

Problem

No Undo Capability

Direct calls leave no trace. Implementing undo means adding bookkeeping fields to every actor class. Every new action doubles the complexity.

Problem

Can't Queue or Schedule

A method call executes immediately. You can't defer it, reorder it, or run it at a specific game tick. Turn-based scheduling becomes impossible.

Problem

Can't Log or Replay

Once the method returns, the action is gone. You can't record what happened, replay a battle, or build a death recap cinematic.

Problem

Hero Knows Too Much

Adding undo, logging, and macro support to Hero violates Single Responsibility. Hero should fight, not manage a history ledger.

The Core Insight

What if instead of calling hero.attack(enemy) directly, you created an object that represents "attack enemy"? That object can be stored in a list, executed later, undone, replayed, logged, or queued. The method call becomes data.

// step_03

Deep Dive

The Command Pattern has four participants. Understanding each role — and how they relate — is the foundation for everything that follows.

Diagram 1 — The Four Participants

Every Command implementation has the same four moving parts. The pattern separates who decides to act from who carries out the act from what the act is.

Diagram 2 — Undo / Redo Stack

Commands are pushed onto a history stack as they execute. Undo pops from the top and calls undo(). Redo pushes them to a redo stack. This is the same mechanism used in every text editor, Photoshop, and version control system.

Diagram 3 — Command Queue (Turn-Based)

Commands waiting to execute are a queue. The game loop dequeues one per tick. This enables AI planning, networked multiplayer (send commands, not state), and replay files.

// roles_explained
  1. Command (interface) Declares execute() and optionally undo(). This is the abstraction that lets the Invoker treat all actions uniformly — it doesn't care if it's an attack or a spell.
  2. Concrete Command (AttackCommand, SpellCommand, MoveCommand) Implements the interface. Stores all parameters needed to perform AND reverse the action. It delegates the actual work to the Receiver — it doesn't contain game logic itself.
  3. Invoker (InputHandler / GameEngine) Calls execute() on commands and manages the history stack. It never calls hero.attack() directly. It just calls command.execute() — it doesn't know or care what the command does.
  4. Receiver (Hero, Enemy) The object that actually carries out the work. AttackCommand.execute() calls hero.attack(). The Receiver has no knowledge of commands at all — it just has its normal methods.
  5. Why store state in the command? AttackCommand stores the target and the damage dealt. When undo() is called, it has everything it needs to reverse the action — no global undo state, no fields polluting Hero.
The Key Transformation

hero.attack(enemy) — a verb, gone when it returns.
new AttackCommand(hero, enemy) — a noun, an object you can store, pass, delay, replay, or undo. This transformation from verb to noun is the entire pattern.

Refactored — Full Command System

The same Hero and Enemy classes from the baseline — untouched. All new code lives in commands and the invoker.

Command.java — The InterfaceJava
/**
 * The Command interface. Every game action implements this.
 * undo() is optional — commands that can't be undone can throw
 * UnsupportedOperationException or simply no-op.
 */
public interface Command {
    void execute();
    void undo();
    default String describe() { return getClass().getSimpleName(); }
}
AttackCommand.javaJava
public class AttackCommand implements Command {

    private final Hero  hero;
    private final Enemy enemy;
    private int damageDealt; // stored AFTER execute — needed by undo()

    public AttackCommand(Hero hero, Enemy enemy) {
        this.hero  = hero;
        this.enemy = enemy;
    }

    @Override
    public void execute() {
        // Delegate the REAL work to the Receiver (Hero)
        damageDealt = hero.getAttackPower();
        enemy.takeDamage(damageDealt);
        System.out.println("[EXEC] Attack → " + damageDealt + " dmg");
    }

    @Override
    public void undo() {
        // Reverse exactly what execute() did
        enemy.heal(damageDealt);
        System.out.println("[UNDO] Attack reversed — healed " + damageDealt);
    }

    @Override
    public String describe() { return "Attack(" + damageDealt + " dmg)"; }
}
SpellCommand.javaJava
public class SpellCommand implements Command {

    private static final int SPELL_DAMAGE = 30;
    private static final int SPELL_COST   = 20;

    private final Hero  hero;
    private final Enemy enemy;
    private boolean executed = false; // guard against undo before execute

    public SpellCommand(Hero hero, Enemy enemy) {
        this.hero = hero; this.enemy = enemy;
    }

    @Override
    public void execute() {
        if (hero.getMp() < SPELL_COST) {
            System.out.println("[EXEC] Not enough MP!");
            return;
        }
        hero.spendMp(SPELL_COST);
        enemy.takeDamage(SPELL_DAMAGE);
        executed = true;
        System.out.println("[EXEC] Fireball → 30 dmg, MP -20");
    }

    @Override
    public void undo() {
        if (!executed) return; // never ran — nothing to undo
        hero.restoreMp(SPELL_COST);
        enemy.heal(SPELL_DAMAGE);
        executed = false;
        System.out.println("[UNDO] Fireball reversed");
    }
}

GameEngine.java — The InvokerJava
import java.util.*;

public class GameEngine {

    private final Deque<Command> history  = new ArrayDeque<>();
    private final Deque<Command> redoStack = new ArrayDeque<>();

    // Execute a command and record it
    public void execute(Command cmd) {
        cmd.execute();
        history.push(cmd);
        redoStack.clear(); // new action clears redo branch
    }

    // Undo the most recent command
    public void undo() {
        if (history.isEmpty()) { System.out.println("Nothing to undo"); return; }
        Command cmd = history.pop();
        cmd.undo();
        redoStack.push(cmd);
    }

    // Redo the most recently undone command
    public void redo() {
        if (redoStack.isEmpty()) { System.out.println("Nothing to redo"); return; }
        Command cmd = redoStack.pop();
        cmd.execute();
        history.push(cmd);
    }

    // Print the full battle log
    public void printHistory() {
        System.out.println("=== Battle Log ===");
        new ArrayList<>(history).stream()
            .map(Command::describe)
            .forEach(System.out::println);
    }
}

ApprenticeSolution 1 — HealCommand

Capture exact heal amount for precise undo.

HealCommand.javaJava
public class HealCommand implements Command {

    private static final int HEAL_AMOUNT = 25;
    private static final int MAX_HP      = 100;

    private final Hero hero;
    private int actualHealed; // captured during execute()

    public HealCommand(Hero hero) { this.hero = hero; }

    @Override
    public void execute() {
        int before = hero.getHp();
        int after  = Math.min(MAX_HP, before + HEAL_AMOUNT);
        actualHealed = after - before; // might be less than 25!
        hero.setHp(after);
        System.out.println("[EXEC] Heal → +" + actualHealed + " HP");
    }

    @Override
    public void undo() {
        hero.setHp(hero.getHp() - actualHealed); // precise reversal
        System.out.println("[UNDO] Heal reversed → -" + actualHealed + " HP");
    }

    @Override
    public String describe() { return "Heal(+" + actualHealed + ")"; }
}

// Key design decision: we store 'actualHealed', not HEAL_AMOUNT.
// If hero was at 90 HP, actualHealed = 10, not 25.
// undo() subtracts 10, restoring hero to 90. ✓
// Storing 25 would undo too much, leaving hero at 75 HP. ✗
ApprenticeSolution 2 — Undo Depth Limit

Enforce max stack size by dropping the oldest entry.

GameEngine.java — bounded historyJava
private static final int MAX_HISTORY = 5;

public void execute(Command cmd) {
    cmd.execute();
    history.push(cmd); // push to front (top of stack)
    if (history.size() > MAX_HISTORY) {
        history.removeLast(); // drop OLDEST (bottom) — it's non-undoable now
    }
    redoStack.clear();
}

// Deque mental model for this use case:
//   push(x)       → adds to FRONT (top of stack)
//   pop()         → removes from FRONT (undo most recent)
//   removeLast()  → removes from BACK (drop oldest)
//
// After 6 commands with limit 5:
//   [Cmd6, Cmd5, Cmd4, Cmd3, Cmd2]  ← Cmd1 dropped
// Undo 5 times works. Cmd1 is gone forever.

VeteranSolution 4 — MacroCommand

Composite of commands — executes and undoes as one unit.

MacroCommand.javaJava
import java.util.*;

public class MacroCommand implements Command {

    private final String        name;
    private final List<Command> commands;

    public MacroCommand(String name, Command... cmds) {
        this.name     = name;
        this.commands = List.of(cmds);
    }

    @Override
    public void execute() {
        System.out.println("[MACRO] Executing: " + name);
        commands.forEach(Command::execute);
    }

    @Override
    public void undo() {
        // Reverse order — undo last step first
        System.out.println("[MACRO] Undoing: " + name);
        List<Command> reversed = new ArrayList<>(commands);
        Collections.reverse(reversed);
        reversed.forEach(Command::undo);
    }

    @Override
    public String describe() { return "Macro(" + name + ")"; }
}

// engine.execute(new MacroCommand("Storm Combo",
//     new AttackCommand(hero, enemy),
//     new MoveCommand(hero, 1, 0),
//     new SpellCommand(hero, enemy),
//     new AttackCommand(hero, enemy)
// ));
//
// engine.undo(); // undoes all 4 steps in reverse order ✓
//
// Design note: pushed as ONE entry on history stack.
// One Ctrl+Z undoes the entire combo — matching user expectation.
VeteranSolution 5 — Turn-Based Action Queue

Schedule actions; process one per turn.

GameEngine.java — with action queueJava
private final Queue<Command> actionQueue = new LinkedList<>();

public void scheduleCommand(Command cmd) {
    actionQueue.offer(cmd); // enqueue — doesn't execute yet
    System.out.println("[QUEUE] Scheduled: " + cmd.describe());
}

public void processTurn() {
    Command cmd = actionQueue.poll(); // dequeue next
    if (cmd == null) {
        System.out.println("[TURN] No actions queued.");
        return;
    }
    execute(cmd); // execute + record in history
    System.out.println("[TURN] Processed. Queue remaining: " + actionQueue.size());
}

// Usage — plan 5 actions, execute one per turn:
// engine.scheduleCommand(new AttackCommand(hero, enemy));
// engine.scheduleCommand(new MoveCommand(hero, 2, 0));
// engine.scheduleCommand(new SpellCommand(hero, enemy));
//
// for (int i = 0; i < 5; i++) engine.processTurn();
//
// Key: scheduleCommand() never executes.
// Only processTurn() executes — and only one per tick.
// AI systems schedule their entire turn plan upfront. ✓
VeteranSolution 6 — Conditional Undo (StunAttack)

Undo must reason about current state, not blindly reverse.

StunAttackCommand.javaJava
public class StunAttackCommand implements Command {

    private final Hero  hero;
    private final Enemy enemy;
    private boolean stunApplied = false;
    private int damageDealt;

    public StunAttackCommand(Hero hero, Enemy enemy) {
        this.hero = hero; this.enemy = enemy;
    }

    @Override
    public void execute() {
        damageDealt = 10;
        enemy.takeDamage(damageDealt);
        if (enemy.isAlive()) {
            enemy.setStunned(true);
            stunApplied = true;
        }
        System.out.println("[EXEC] StunAttack: 10 dmg, stunned=" + stunApplied);
    }

    @Override
    public void undo() {
        // Only heal if enemy is still alive — don't resurrect
        if (enemy.isAlive()) {
            enemy.heal(damageDealt);
        }
        // Always un-stun if we applied it
        if (stunApplied) {
            enemy.setStunned(false);
            stunApplied = false;
        }
        System.out.println("[UNDO] StunAttack reversed (alive=" + enemy.isAlive() + ")");
    }
}

// Lesson: undo doesn't always mean "perfect reversal".
// Real undo systems have constraints:
//   - Can't un-send an email
//   - Can't un-delete a file if disk space was reclaimed
//   - Can't resurrect a dead enemy (game balance reasons)
// Document what your undo does and doesn't reverse.
VeteranSolution 7 — Replay Mode

Decouple commands from specific receiver instances.

Approach — Command Factory / Supplier patternJava
// Problem: commands hold references to the original hero/enemy.
// For replay on new objects, we need a way to re-create them
// targeting the new receivers.
//
// Solution A — Command Factories (Suppliers):
// Instead of storing commands, store Supplier<Command> lambdas.
// The lambda captures the action, not the receivers.
// For replay, invoke the supplier with new receivers.

private final List<BiFunction<Hero, Enemy, Command>> replayScript = new ArrayList<>();

public void execute(Command cmd, BiFunction<Hero, Enemy, Command> factory) {
    cmd.execute();
    history.push(cmd);
    replayScript.add(factory); // store HOW to recreate, not the instance
    redoStack.clear();
}

public void replay() {
    Hero  freshHero  = new Hero();
    Enemy freshEnemy = new Enemy("Dragon");
    System.out.println("=== REPLAY ===");
    List<BiFunction<Hero,Enemy,Command>> steps = new ArrayList<>(replayScript);
    Collections.reverse(steps); // history is LIFO, replay needs FIFO
    steps.forEach(f -> f.apply(freshHero, freshEnemy).execute());
}

// Usage:
// engine.execute(
//   new AttackCommand(hero, enemy),
//   (h, e) -> new AttackCommand(h, e)  // factory for replay
// );
//
// engine.replay(); // fresh hero/enemy, same sequence ✓
//
// Solution B (simpler): give each Command a setReceivers() method
// so replay can rebind after deserialisation.

// All Command implementations must implement Serializable:
public class AttackCommand implements Command, Serializable { ... }

// Serialising the history:
public void saveLog(String path) throws IOException {
    try (var oos = new ObjectOutputStream(new FileOutputStream(path))) {
        oos.writeObject(new ArrayList<>(history));
    }
}

// Reading back and replaying:
@SuppressWarnings("unchecked")
public void replayFromLog(String path, Hero hero, Enemy enemy) throws Exception {
    try (var ois = new ObjectInputStream(new FileInputStream(path))) {
        List<Command> log = (List<Command>) ois.readObject();
        Collections.reverse(log); // restore chronological order
        log.forEach(cmd -> {
            cmd.rebind(hero, enemy); // re-wire receivers post-deserialise
            cmd.execute();
        });
    }
}

@FunctionalInterface
public interface CommandExecutionListener {
    void onExecute(Command cmd);
    default void onUndo(Command cmd) {}
}

// In GameEngine:
private final List<CommandExecutionListener> listeners = new ArrayList<>();

public void addListener(CommandExecutionListener l) { listeners.add(l); }

public void execute(Command cmd) {
    cmd.execute();
    history.push(cmd);
    redoStack.clear();
    listeners.forEach(l -> l.onExecute(cmd)); // notify observers
}

public void undo() {
    if (history.isEmpty()) return;
    Command cmd = history.pop();
    cmd.undo();
    redoStack.push(cmd);
    listeners.forEach(l -> l.onUndo(cmd)); // notify observers
}

// Receiver
public class TextBuffer {
    private final StringBuilder buf = new StringBuilder();

    public void insert(int pos, String text) { buf.insert(pos, text); }
    public void delete(int pos, int len)    { buf.delete(pos, pos + len); }
    public String get()                      { return buf.toString(); }
}

// Insert command
public class InsertTextCommand implements Command {
    private final TextBuffer buf;
    private final int         pos;
    private final String     text;

    public InsertTextCommand(TextBuffer buf, int pos, String text) {
        this.buf = buf; this.pos = pos; this.text = text;
    }

    @Override public void execute() { buf.insert(pos, text); }
    @Override public void undo()    { buf.delete(pos, text.length()); }
}

// Delete command
public class DeleteTextCommand implements Command {
    private final TextBuffer buf;
    private final int         pos, len;
    private String deleted; // captured during execute for undo

    public DeleteTextCommand(TextBuffer buf, int pos, int len) {
        this.buf = buf; this.pos = pos; this.len = len;
    }

    @Override
    public void execute() {
        deleted = buf.get().substring(pos, pos + len); // save what we delete
        buf.delete(pos, len);
    }

    @Override
    public void undo() { buf.insert(pos, deleted); } // re-insert exactly
}

Real-World Connections

IntelliJ / VSCode: Every keystroke, deletion, and refactor is a Command. Ctrl+Z pops the history stack. Ctrl+Y re-executes from the redo stack.

Git: Commits are commands. git revert creates a new undo command. git cherry-pick replays a command on a different branch.

Networking / RPC: Remote calls are serialised commands sent over the wire. The server dequeues and executes them. This is the foundation of event sourcing.

Spring Batch: Jobs are composed of Steps, which are Command objects. Failed jobs can be retried (redo) or rolled back (undo) at the step level.
