Editor.java — direct calls, no command pattern yetJava
public class Editor {

    private final TextBuffer       buffer    = new TextBuffer();
    private final FormattingEngine formatter = new FormattingEngine();
    private final SelectionModel   selection = new SelectionModel();

    public void type(String text) {
        buffer.insert(selection.getCursor(), text);
        selection.moveCursor(selection.getCursor() + text.length());
    }

    public void deleteSelection() {
        if (!selection.hasSelection()) return;
        buffer.delete(selection.getSelectionStart(), selection.getSelectionEnd());
        selection.moveCursor(selection.getSelectionStart());
        selection.clearSelection();
    }

    public void boldSelection() {
        if (!selection.hasSelection()) return;
        formatter.applyBold(selection.getSelectionStart(), selection.getSelectionEnd());
    }

    public void replaceAll(String find, String replace) {
        String current = buffer.getText();
        buffer.clear();
        buffer.insert(0, current.replace(find, replace));
    }

    // ⚠ No undo. No redo. No history.
    // type() is called — the text is added. That operation is gone.
    // Ctrl+Z would need to know what was just typed, where, how much.
    // None of that is stored anywhere.

    public String getContent() { return buffer.getText(); }
}
Works — For Now

A real intermediate editor. The buffer, formatter, and selection model are all proper objects. The problem isn't the data model — it's that every operation is a fire-and-forget method call. There's nothing to press Ctrl+Z on.

New Requirements That Break It

Every real editor needs undo. Beyond that: redo, operation history, collaborative editing (send operations over the wire, not the full document), macros, and autosave checkpoints. None of these are possible when operations are plain method calls.

New Requirements

① Ctrl+Z / Ctrl+Y — unlimited undo and redo. ② Operation log — show users what has happened ("Typed 'hello'", "Deleted 3 chars"). ③ Batch operations — Find & Replace changes 47 occurrences; undo should reverse all 47 at once. ④ Collaborative editing — send operations as serialisable objects to another user's editor. ⑤ Macro recording — record a sequence, replay with one keystroke.

❌ Attempting undo with direct calls — impossible without restructuringJava

Violation

No History Depth

The "last action" approach only undoes one step. Users expect to undo arbitrarily far back. Every field in Editor only tracks the most recent operation.

Violation

Editor Knows Too Much

Editor is now responsible for text editing, formatting, selection, undo bookkeeping, and operation dispatch. SRP is gone. Every new operation adds more fields to an already bloated class.

Violation

Batch Undo Is Impossible

Find & Replace modifies the buffer in one call. There's no way to track each individual substitution to undo them. Only a full snapshot works — expensive and non-composable.

Violation

No Network / Macro Support

Collaborative editing requires sending operations over the network. Method calls can't be serialised or replayed on another machine. Macros need a list of recorded operations — you can't store method calls.

The Core Insight

Turn each edit operation from a verb (a method call that fires and disappears) into a noun (an object that can be stored, reversed, replayed, and sent over the wire). buffer.insert(5, "hello") becomes new InsertCommand(buffer, 5, "hello") — an object that carries everything needed to execute and undo the action.

Deep Dive

The Command Pattern in a text editor is particularly rich: undo/redo stacks, batch commands, coalescing (merging adjacent keystrokes), and the relationship between commands and the document model are all important real-world concerns.

Diagram 1 — Command Pattern Structure

Four participants. The Invoker (EditorEngine) manages the history stack. The Command interface abstracts all operations. Concrete Commands store the data needed to execute and reverse. The Receiver (TextBuffer, FormattingEngine) does the actual work.

Diagram 2 — Undo / Redo Stack Mechanics

The history stack is the heart of any editor's undo system. Three rules govern it: execute pushes to history, undo pops from history and pushes to redo, and any new command clears the redo stack (you can't redo after diverging).

Diagram 3 — Command Coalescing

Users type characters one at a time but expect Ctrl+Z to undo whole words, not individual letters. Coalescing merges adjacent compatible commands into one. Every text editor implements this — IntelliJ, VSCode, and Word all coalesce keystrokes into word-level undo units.

// deep_dive_topics
  1. What state must an InsertCommand store?The position and the text are obvious. But undo() also needs to restore the cursor position. If the user typed at position 5 and the cursor moved to 10, undo() must move the cursor back to 5. Store everything that changed — buffer state, cursor, selection.
  2. What state must a DeleteCommand store?The deleted text — because it's gone once the deletion executes. execute() must capture the text being deleted before deleting it. This is why commands are objects not method calls: they can run pre-execution capture logic.
  3. Coalescing rule: when should adjacent commands merge?Two InsertCommands coalesce if: same type, the new insertion starts exactly where the previous one ends (contiguous), and they occur within a time threshold (typically 400–1000ms). A space or punctuation character usually breaks the coalesce window — that's why Ctrl+Z undoes word by word.
  4. Batch commands vs coalescingCoalescing merges commands silently after the fact. A BatchCommand is an explicit composite — Find & Replace creates one BatchCommand wrapping N ReplaceCommand instances. Undo reverses all N at once because they are a single unit in the history stack.
  5. Why clearing redo on execute is correctAfter undoing 3 steps and then typing a new character, the future you undid no longer exists. Keeping the redo stack would allow branching history — possible but complex (see: version control). Simple editors always clear redo on new command.
  6. Serialisable commands for collaborative editingIf InsertCommand implements Serializable (or serialises to JSON), it can be sent to another user's editor and applied. Operational Transformation (OT) and CRDT systems — the technology behind Google Docs — are built on serialisable command objects. Your history stack becomes a distributed event log.
Text Editor = Canonical Command Pattern Domain

The text editor is not just an example of the Command Pattern — it is the canonical example. The Gang of Four book introduces Command with a text editor. IntelliJ's entire undo system is a public API (com.intellij.openapi.command.CommandProcessor). VSCode's undo is built on IUndoRedoElement. Every time you press Ctrl+Z in any editor, you are watching the Command Pattern execute.

Command vs Memento — When to Use Each

Both patterns support undo. Choosing the wrong one is a common mistake.

Aspect Command Pattern Memento Pattern
Undo strategy Reverse the operation (insert → delete) Restore a snapshot of the whole state
Memory cost Low — only store what changed (the text delta) High — full copy of document per step
Redo support Natural — re-execute the command Requires saving forward snapshots too
Network / collab Commands are serialisable operations Snapshots can't be merged across clients
Best for Text editors, CAD tools, databases Games (save states), config rollback
Complexity Each command must implement reverse logic Simple — just copy/restore state

EditorCommand.java — The InterfaceJava
/**
 * Every editor operation implements this.
 * describe() powers the history panel ("Typed 'hello'", "Deleted 3 chars").
 * canCoalesceWith() enables word-level undo by merging adjacent keystrokes.
 */
public interface EditorCommand {
    void execute();
    void undo();

    default String  describe()                          { return getClass().getSimpleName(); }
    default boolean canCoalesceWith(EditorCommand other) { return false; }
    default void    coalesceWith(EditorCommand other)    { }
}
InsertCommand.javaJava
public class InsertCommand implements EditorCommand {

    private final TextBuffer     buffer;
    private final SelectionModel selection;
    private final int           position;
    private       String        text;          // mutable: grows during coalescing
    private final int           prevCursor;    // cursor before — needed by undo()
    private final long          timestamp;     // for coalesce time window

    public InsertCommand(TextBuffer buf, SelectionModel sel, int pos, String text) {
        this.buffer    = buf;
        this.selection = sel;
        this.position  = pos;
        this.text      = text;
        this.prevCursor = sel.getCursor();
        this.timestamp  = System.currentTimeMillis();
    }

    @Override
    public void execute() {
        buffer.insert(position, text);
        selection.moveCursor(position + text.length());
    }

    @Override
    public void undo() {
        buffer.delete(position, position + text.length());
        selection.moveCursor(prevCursor); // restore cursor exactly
    }

    // Coalescing: merge if both are inserts, positions are contiguous, and within 800ms
    @Override
    public boolean canCoalesceWith(EditorCommand other) {
        if (!(other instanceof InsertCommand o)) return false;
        return o.position == position + text.length()  // contiguous
            && o.timestamp - timestamp < 800;           // within time window
    }

    @Override
    public void coalesceWith(EditorCommand other) {
        text += ((InsertCommand) other).text; // grow the text — no new stack entry
    }

    @Override
    public String describe() {
        String preview = text.length() > 20 ? text.substring(0, 20) + "…" : text;
        return "Typed \"" + preview + "\"";
    }
}
DeleteCommand.javaJava
public class DeleteCommand implements EditorCommand {

    private final TextBuffer     buffer;
    private final SelectionModel selection;
    private final int           start, end;
    private       String        deletedText; // captured BEFORE execute()
    private final int           prevCursor;

    public DeleteCommand(TextBuffer buf, SelectionModel sel, int start, int end) {
        this.buffer     = buf;
        this.selection  = sel;
        this.start      = start;
        this.end        = end;
        this.prevCursor = sel.getCursor();
    }

    @Override
    public void execute() {
        deletedText = buffer.substring(start, end); // capture BEFORE deleting!
        buffer.delete(start, end);
        selection.moveCursor(start);
        selection.clearSelection();
    }

    @Override
    public void undo() {
        buffer.insert(start, deletedText); // re-insert exactly what was removed
        selection.moveCursor(prevCursor);
    }

    @Override
    public String describe() {
        return "Deleted " + (end - start) + " char(s)";
    }
}
FormatCommand.java + BatchCommand.javaJava
public class FormatCommand implements EditorCommand {

    private final FormattingEngine formatter;
    private final FormatType      type;
    private final int             start, end;
    private       FormatSpan     appliedSpan; // stored for undo()

    public FormatCommand(FormattingEngine f, FormatType type, int start, int end) {
        this.formatter = f; this.type = type;
        this.start = start; this.end = end;
    }

    @Override
    public void execute() {
        appliedSpan = formatter.applyFormat(type, start, end); // returns the span created
    }

    @Override
    public void undo() {
        formatter.removeSpan(appliedSpan); // removes exactly the span we added
    }

    @Override
    public String describe() { return "Applied " + type + " (" + start + "-" + end + ")"; }
}

// BatchCommand — executes N commands as a single undo unit
// Used by: Find&Replace, paste, multi-cursor edits
// ─────────────────────────────────────────────────────────
public class BatchCommand implements EditorCommand {

    private final String              label;
    private final List<EditorCommand> commands;

    public BatchCommand(String label, List<EditorCommand> commands) {
        this.label    = label;
        this.commands = List.copyOf(commands);
    }

    @Override public void execute() { commands.forEach(EditorCommand::execute); }

    @Override
    public void undo() {
        var reversed = new ArrayList<>(commands);
        Collections.reverse(reversed);
        reversed.forEach(EditorCommand::undo);
    }

    @Override public String describe() { return label + " (" + commands.size() + " ops)"; }
}
EditorEngine.java — The InvokerJava
public class EditorEngine {

    private final Deque<EditorCommand> history   = new ArrayDeque<>();
    private final Deque<EditorCommand> redoStack = new ArrayDeque<>();
    private static final int MAX_HISTORY = 200;

    public void execute(EditorCommand cmd) {
        cmd.execute();

        // Try to coalesce with the top of the history stack
        if (!history.isEmpty() && history.peek().canCoalesceWith(cmd)) {
            history.peek().coalesceWith(cmd); // merge into existing entry
        } else {
            history.push(cmd);              // add new entry
            if (history.size() > MAX_HISTORY) history.removeLast();
        }
        redoStack.clear();
    }

    public void undo() {
        if (history.isEmpty()) { System.out.println("Nothing to undo"); return; }
        EditorCommand cmd = history.pop();
        cmd.undo();
        redoStack.push(cmd);
        System.out.println("[UNDO] " + cmd.describe());
    }

    public void redo() {
        if (redoStack.isEmpty()) { System.out.println("Nothing to redo"); return; }
        EditorCommand cmd = redoStack.pop();
        cmd.execute();
        history.push(cmd);
        System.out.println("[REDO] " + cmd.describe());
    }

    public List<String> getHistoryLog() {
        return history.stream().map(EditorCommand::describe).toList();
    }
}

DraftChallenge 1

ReplaceAllCommand

Create a ReplaceAllCommand that performs a find-and-replace across the entire TextBuffer. execute() substitutes every occurrence of the search string. undo() must perfectly restore the original text — even when there are 0 matches. Store the minimum necessary state: you do not need to snapshot the entire document.

Think About

The naive approach is to snapshot the whole buffer before replacement. But that's O(n) memory per operation. A smarter approach records each individual substitution as an offset + replaced text. Which is more appropriate here, and why?

DraftChallenge 2

History Panel

Add a getHistoryPanel() method to EditorEngine that returns a formatted list of all history entries, newest first, numbered from 1. Each entry shows the step number, description, and an indicator of which step is "current" (the most recently executed). This is the history list shown in IntelliJ's Local History.

Goal

Implement describe() properly on all commands — this is the method that surfaces in the UI. Good descriptions ("Typed 'hello world'") are what separate professional editors from toys.

DraftChallenge 3

Undo Checkpoint / Save Point

Add a setSavePoint() method to EditorEngine that marks the current history position. Add isModifiedSinceSave() that returns true if any commands have been executed or undone since the save point. This is how editors show the "unsaved changes" indicator (the dot in the VSCode tab title).

Think About

The save point is a reference to a position in the history stack — not a snapshot of content. If the user undoes back to exactly the save point, isModifiedSinceSave() should return false again. How do you track "same position" without snapshots?

EditorChallenge 4

Word-Level Undo via Coalescing

Extend the coalescing logic in InsertCommand: break the coalesce window whenever a space, newline, or punctuation character is typed. Each "word" typed should coalesce into a single undo step, but whitespace starts a new step. Write a test that types "Hello world foo" character by character and verifies the history stack has exactly three entries: "Hello", " world", " foo".

EditorChallenge 5

Cursor-Aware Paste Command

Create a PasteCommand that inserts the contents of a simulated clipboard (String clipboard) at the current cursor position. If a selection is active, the selection is replaced (implement this as a BatchCommand of a DeleteCommand + InsertCommand). Undo must correctly restore the original selection state — including the selection range, not just the cursor position.

Think About

Paste-with-selection is a compound operation — it's naturally a BatchCommand. But should it appear in the history as one entry ("Pasted 'hello world'") or two ("Deleted selection", "Inserted 'hello world'")? User expectation says one. How do you reconcile that with a BatchCommand?

EditorChallenge 6

Macro Recorder

Add startRecording() and stopRecording(): MacroCommand to EditorEngine. While recording, all executed commands are captured into a list. stopRecording() returns a MacroCommand that wraps the captured list. Replaying the macro re-executes all commands. The macro itself should be executable and undoable as a single unit.

Think About

Should the recorded commands be the same instances that were executed during recording, or copies? If they are the same instances, replaying the macro calls execute() on already-executed commands. Is that safe? What happens to state captured during the original execution?

EditorChallenge 7

TransactionalCommand — All-or-Nothing

Create a TransactionalCommand that executes a list of sub-commands. If any sub-command throws an exception during execute(), all previously-executed sub-commands in the batch are automatically undone before the exception propagates. The history stack should never contain a partially-applied batch.

Why This Matters

This is ACID atomicity applied to editor commands. Database transactions work the same way: if any statement fails, all changes are rolled back. The pattern is identical — execute, catch failure, undo completed steps in reverse order.

ArchitectChallenge 8

Serialisable Command Log — Crash Recovery

Make InsertCommand, DeleteCommand, and FormatCommand implement Serializable. After every execute(), append the command to a commands.log file. On startup, if the log file exists, replay all commands in order to restore the document to its last known state — even if the editor crashed. This is how write-ahead logs in databases provide crash recovery.

Think About

The log grows forever. When should it be truncated? On a clean save, the log can be cleared — the document file is now the ground truth. How do you handle the case where the crash log exists but the document file was also saved partway? Which is more recent?

ArchitectChallenge 9

Operational Transformation — Basic Collaborative Editing

Two editors (User A and User B) each have a copy of the same document. User A inserts "hello" at position 0. Simultaneously, User B inserts "world" at position 0. When A's command arrives at B's editor, it must be transformed against B's already-applied command before being applied — otherwise positions are wrong. Implement a transform(InsertCommand incoming, InsertCommand applied) method that adjusts the incoming command's position based on the already-applied one.

Why This Matters

This is Operational Transformation — the core algorithm behind Google Docs, Etherpad, and CRDTs. It's a direct extension of the Command Pattern: commands are serialisable, transformable, and replayable. Every Google Doc edit is a transformed Command object.

BOSSChallenge 10

New Domain — Database Migration Runner

Port Command to database migrations. Receiver: a Database class with executeSQL(String). Commands: CreateTableCommand, AddColumnCommand, SeedDataCommand. DraftSolution 1 — ReplaceAllCommand

Track individual substitutions, not a full snapshot.

ReplaceAllCommand.javaJava
public class ReplaceAllCommand implements EditorCommand {

    private final TextBuffer buffer;
    private final String     find, replace;

    // Records each substitution: offset where it occurred + original text there
    // Much cheaper than snapshotting the whole buffer
    private record Substitution(int offset, String original, String replacement) {}
    private final List<Substitution> subs = new ArrayList<>();

    public ReplaceAllCommand(TextBuffer buf, String find, String replace) {
        this.buffer = buf; this.find = find; this.replace = replace;
    }

    @Override
    public void execute() {
        subs.clear();
        String text = buffer.getText();
        int offset = 0;
        int idx;
        // Walk through finding each occurrence, tracking live offsets
        while ((idx = text.indexOf(find, offset)) != -1) {
            subs.add(new Substitution(idx, find, replace));
            text = text.substring(0, idx) + replace + text.substring(idx + find.length());
            offset = idx + replace.length();
        }
        buffer.clear();
        buffer.insert(0, text);
    }

    @Override
    public void undo() {
        // Reverse substitutions in reverse order to keep offsets valid
        String text = buffer.getText();
        List<Substitution> reversed = new ArrayList<>(subs);
        Collections.reverse(reversed);
        for (Substitution s : reversed) {
            text = text.substring(0, s.offset())
                 + s.original()
                 + text.substring(s.offset() + s.replacement().length());
        }
        buffer.clear();
        buffer.insert(0, text);
    }

    @Override
    public String describe() {
        return "Replace all \"" + find + "\"→\"" + replace + "\" (" + subs.size() + " occurrences)";
    }
}

// When zero matches: subs is empty. undo() is a no-op. ✓
// Memory: O(k) where k = number of matches. Beats O(n) snapshot. ✓
// Note: if find and replace have different lengths, offsets shift.
//   That's why we reverse the substitutions on undo — rightmost first.

public String getHistoryPanel() {
    if (history.isEmpty()) return "(no history)";
    var sb = new StringBuilder();
    var entries = new ArrayList<>(history); // snapshot (newest first)
    for (int i = 0; i < entries.size(); i++) {
        String marker = (i == 0) ? " ← current" : "";
        sb.append(String.format("  %2d. %s%s%n", i + 1, entries.get(i).describe(), marker));
    }
    return sb.toString();
}

DraftSolution 3 — Undo Checkpoint

Track history depth at save time, compare on each change.

EditorEngine.java — save point trackingJava
// Track the command that was at the top of history when saved
// null = never saved (modified from the start)
private EditorCommand savePointCommand = null;
private boolean        savedOnEmptyHistory = false;

public void setSavePoint() {
    savePointCommand    = history.isEmpty() ? null : history.peek();
    savedOnEmptyHistory = history.isEmpty();
}

public boolean isModifiedSinceSave() {
    if (savedOnEmptyHistory) return !history.isEmpty();
    if (savePointCommand == null) return !history.isEmpty();
    // Modified if the top of history is no longer the saved command
    return history.isEmpty() || history.peek() != savePointCommand;
}

private static final String WORD_BOUNDARIES = " \t\n.,;:!?()[]{}\"'";

@Override
public boolean canCoalesceWith(EditorCommand other) {
    if (!(other instanceof InsertCommand o)) return false;
    // Break coalesce if THIS command's text ends with a word boundary
    // (i.e. the user just typed a space — next char starts new word)
    if (!text.isEmpty() && WORD_BOUNDARIES.indexOf(text.charAt(text.length()-1)) >= 0)
        return false;
    // Also break if the INCOMING char is a word boundary
    if (!o.text.isEmpty() && WORD_BOUNDARIES.indexOf(o.text.charAt(0)) >= 0)
        return false;
    return o.position == position + text.length()  // contiguous
        && o.timestamp - timestamp < 800;           // time window
}

// Test verification:
// Type: H-e-l-l-o (coalesce) → "Hello" [1 history entry]
// Type: space → breaks coalesce window
// Type: w-o-r-l-d (coalesce) → " world" [new entry, includes space]
// Type: space
// Type: f-o-o → " foo" [new entry]
//
// history.size() == 3: ["Hello", " world", " foo"] ✓
// Ctrl+Z: "foo" → Ctrl+Z: "world" → Ctrl+Z: "Hello" → empty ✓
// This matches VSCode/IntelliJ word-level undo exactly.
EditorSolution 5 — Paste Command

Paste is a BatchCommand internally, but appears as one undo step.

PasteCommand.javaJava
public class PasteCommand implements EditorCommand {

    private final TextBuffer     buffer;
    private final SelectionModel selection;
    private final String         clipboard;

    // Inner commands built during execute() — not before, because positions
    // depend on the selection state at execute time
    private EditorCommand inner;

    public PasteCommand(TextBuffer buf, SelectionModel sel, String clipboard) {
        this.buffer = buf; this.selection = sel; this.clipboard = clipboard;
    }

    @Override
    public void execute() {
        if (selection.hasSelection()) {
            // Delete selection + insert: two ops, one undo unit
            var del = new DeleteCommand(buffer, selection,
                                       selection.getSelectionStart(),
                                       selection.getSelectionEnd());
            var ins = new InsertCommand(buffer, selection,
                                       selection.getSelectionStart(), clipboard);
            inner = new BatchCommand("Paste", List.of(del, ins));
        } else {
            inner = new InsertCommand(buffer, selection,
                                        selection.getCursor(), clipboard);
        }
        inner.execute();
    }

    @Override
    public void undo()    { inner.undo(); }

    @Override
    public String describe() {
        String preview = clipboard.length() > 20
            ? clipboard.substring(0, 20) + "…" : clipboard;
        return "Pasted \"" + preview + "\"";
    }
}

// Key design: PasteCommand builds its inner command(s) lazily at execute() time.
// If built at construction time, the selection might change before execute() runs.
//
// User sees: "Pasted 'hello world'" in the history panel — one clean entry. ✓
// Undo restores both the inserted text AND the original selection. ✓
// inner.undo() calls BatchCommand.undo() which reverses delete+insert in order. ✓

// MacroCommand — replays using FRESH command instances (not originals)
public class MacroCommand implements EditorCommand {
    private final String                       name;
    private final List<Supplier<EditorCommand>> steps; // factories, not instances

    public MacroCommand(String name, List<EditorCommand> recorded) {
        this.name  = name;
        // Store suppliers that create fresh copies on replay
        this.steps = recorded.stream().map(cmd -> (Supplier<EditorCommand>) cmd::copy).toList();
    }

    @Override
    public void execute() { steps.forEach(s -> s.get().execute()); }

    @Override
    public void undo() {
        var all = steps.stream().map(Supplier::get).toList();
        var rev = new ArrayList<>(all);
        Collections.reverse(rev);
        rev.forEach(EditorCommand::undo);
    }

    public int size() { return steps.size(); }
    @Override public String describe() { return "Macro '" + name + "' (" + steps.size() + " steps)"; }
}

// Why fresh copies on replay (copy() method on each command)?
// Original instances have state set during the first execution (e.g. deletedText).
// Re-running execute() on them would use stale captured state.
// Each replay should capture its own fresh state at replay time.
// Add EditorCommand.copy() as a default method or abstract method. ✓
EditorSolution 7 — TransactionalCommand

Execute, catch failure, undo completed steps in reverse.

TransactionalCommand.javaJava
public class TransactionalCommand implements EditorCommand {

    private final String              label;
    private final List<EditorCommand> commands;
    private final List<EditorCommand> executed = new ArrayList<>();

    public TransactionalCommand(String label, List<EditorCommand> commands) {
        this.label    = label;
        this.commands = commands;
    }

    @Override
    public void execute() {
        executed.clear();
        for (EditorCommand cmd : commands) {
            try {
                cmd.execute();
                executed.add(cmd); // only add AFTER successful execute
            } catch (Exception e) {
                // Rollback: undo all previously executed commands in reverse
                Collections.reverse(new ArrayList<>(executed))
                           .forEach(EditorCommand::undo);
                executed.clear();
                throw new RuntimeException("Transaction failed, rolled back: " + e.getMessage(), e);
            }
        }
    }

    @Override
    public void undo() {
        var rev = new ArrayList<>(executed);
        Collections.reverse(rev);
        rev.forEach(EditorCommand::undo);
    }

    @Override
    public String describe() { return label + " (transactional, " + commands.size() + " ops)"; }
}

// This is ACID atomicity: either all commands apply, or none do.
// The pattern is identical to database transactions:
//   BEGIN TRANSACTION
//   try { execute each statement }
//   catch { ROLLBACK (undo completed statements) }
//   finally { if ok: COMMIT (add to history) }
//
// Flyway uses this — each migration runs in a transaction.
// Spring's @Transactional uses the same logic via AOP proxies.

// Startup recovery:
public static EditorEngine recoverFromLog(TextBuffer buf, SelectionModel sel)
                                             throws Exception {
    File log = new File(LOG_FILE);
    EditorEngine engine = new EditorEngine();
    if (!log.exists()) return engine;

    System.out.println("[RECOVER] Replaying command log...");
    try (var ois = new ObjectInputStream(new FileInputStream(log))) {
        while (true) {
            try {
                EditorCommand cmd = (EditorCommand) ois.readObject();
                cmd.rebind(buf, sel); // re-wire to new receiver instances
                engine.execute(cmd);
            } catch (EOFException eof) { break; }
        }
    }
    System.out.println("[RECOVER] Done. " + engine.history.size() + " commands replayed.");
    return engine;
}

// On clean save: log.delete() — document file is ground truth. ✓
// On crash: log exists → replay all commands from scratch. ✓
// This is PostgreSQL's WAL (Write-Ahead Log) in miniature.
// Every database uses this pattern for crash recovery.
ArchitectSolution 9 — Operational Transformation

Transform incoming commands against already-applied ones before replay.

OperationalTransform.javaJava
public class OperationalTransform {

    /**
     * Transform 'incoming' InsertCommand against an already-applied 'applied' InsertCommand.
     * Returns a new InsertCommand adjusted to produce the correct result.
     *
     * Rule: if applied.position <= incoming.position, the text was inserted before
     * the incoming position, so shift incoming right by applied.length.
     */
    public static InsertCommand transform(
            InsertCommand incoming,
            InsertCommand applied,
            TextBuffer buf, SelectionModel sel) {

        int newPos = incoming.getPosition();
        if (applied.getPosition() <= incoming.getPosition()) {
            newPos += applied.getText().length(); // shift right
        }
        return new InsertCommand(buf, sel, newPos, incoming.getText());
    }
}

// Scenario:
// Initial document: "world"
//
// User A (position 0): inserts "Hello " → sends InsertCommand(0, "Hello ")
// User B (position 0): inserts "foo "   → applies locally first
//
// B's document after local apply: "foo world"
// A's command arrives at B: InsertCommand(0, "Hello ")
// Without OT: inserts at 0 → "Hello foo world"  ✗ (wrong)
//
// With OT: transform A's (pos=0) against B's (pos=0, len=4):
//   applied.pos (0) <= incoming.pos (0) → newPos = 0 + 4 = 4
//   Apply transformed: InsertCommand(4, "Hello ") → "foo Hello world" ✓
//
// Real OT handles Insert vs Delete, Delete vs Delete, Delete vs Insert.
// CRDTs are a more sophisticated approach that avoids the need for
// a central server to order transforms.
// Google Docs used OT from 2006; Wave used OT; Figma uses CRDTs.
// This is Command Pattern + network = distributed editor. ✓

// The Command interface — renamed to match domain
public interface Migration {
    int    version();    // e.g. 1, 2, 3
    String describe();
    void   up(Database db);    // execute()
    void   down(Database db);  // undo()
}

public class CreateUserTableMigration implements Migration {
    @Override public int version()  { return 1; }
    @Override public String describe() { return "Create users table"; }

    @Override
    public void up(Database db) {
        db.execute("CREATE TABLE users (id INT PRIMARY KEY, name VARCHAR(100))");
    }

    @Override
    public void down(Database db) {
        db.execute("DROP TABLE users");
    }
}

// The Invoker — exactly like EditorEngine
public class MigrationRunner {
    private final Database       db;
    private final Deque<Migration> applied = new ArrayDeque<>();

    public MigrationRunner(Database db) { this.db = db; }

    public void migrate(Migration m) {
        m.up(db);
        applied.push(m);
        System.out.println("[MIGRATE] V" + m.version() + ": " + m.describe());
    }

    public void rollback() {
        if (applied.isEmpty()) { System.out.println("Nothing to rollback"); return; }
        Migration m = applied.pop();
        m.down(db);
        System.out.println("[ROLLBACK] V" + m.version() + ": " + m.describe());
    }

    public void rollbackTo(int targetVersion) {
        while (!applied.isEmpty() && applied.peek().version() > targetVersion) {
            rollback();
        }
    }
}

Real-World Connections

IntelliJ IDEA: The entire undo system is com.intellij.openapi.command.CommandProcessor. Every editor action — typing, refactoring, formatting — is wrapped in a Command. Undo merges adjacent commands into groups (coalescing). The API is public and plugins use it directly.

VSCode: IUndoRedoElement is the command interface. The undo stack is per-resource (per file). Undo groups (UndoRedoGroup) are BatchCommands. Language extensions contribute their own undoable edits via the TextEdit API.

Google Docs / collaborative editors: Each keystroke is a serialised Command (InsertOperation, DeleteOperation). Operational Transformation (Challenge 9) ensures commands from different users commute correctly. The document is the accumulated result of replaying all commands from an empty buffer.

PostgreSQL WAL (Write-Ahead Log): Every database write is first written as a command record to the WAL before modifying data pages. On crash recovery, the WAL is replayed from the last checkpoint. This is Challenge 8 at petabyte scale.

Flyway / Liquibase: Database migrations are Commands with up() and down(). The migration history table is the history stack. flyway.undo() pops from the stack and calls down(). Challenge 10 exactly.
