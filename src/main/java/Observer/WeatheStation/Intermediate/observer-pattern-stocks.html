public class StockFeed {

    private String  symbol;
    private double  price;
    private double  previousPrice;

    // Hard references to every subscriber — the core problem
    private AlertService  alertService  = new AlertService();
    private Dashboard     dashboard     = new Dashboard();
    private AuditLogger   auditLogger   = new AuditLogger();

    public StockFeed(String symbol, double initialPrice) {
        this.symbol        = symbol;
        this.price         = initialPrice;
        this.previousPrice = initialPrice;
    }

    public void updatePrice(double newPrice) {
        previousPrice = price;
        price         = newPrice;

        // Directly calling every subscriber — tightly coupled
        alertService.checkAlerts(symbol, price, previousPrice);
        dashboard.redraw(symbol, price, previousPrice);
        auditLogger.log(symbol, price, previousPrice);
    }

    public double getPrice()  { return price; }
    public String getSymbol() { return symbol; }
}

Works — For Now

The update logic is solid. The problem is entirely in how StockFeed is connected to its subscribers. Every subscriber is hardwired in — adding a fourth means editing StockFeed. Click ② to see why this becomes unworkable fast.

New Requirements That Break It

The platform grows. A mobile push notification service, a risk engine, a tax reporting module, and a third-party analytics webhook all need price updates. Different clients need different feeds. Some only care about AAPL. Some want all symbols. Some need to subscribe dynamically at runtime.

New Requirements

① A MobileNotifier needs to fire push notifications on 5%+ moves. ② A RiskEngine computes portfolio exposure — needs all symbols, live. ③ A TaxReporter logs realised gains — only subscribe during tax season. ④ Subscribers must be able to register and unregister at runtime without restarting the feed.

Violation

Open/Closed Principle

Every new subscriber requires editing StockFeed. A class that needs to change whenever a new consumer appears is not closed for modification.

Violation

Single Responsibility

StockFeed tracks prices and manages a list of consumers and knows their exact method names. That is three jobs, not one.

Violation

No Runtime Flexibility

TaxReporter is always active even outside tax season. You can't subscribe or unsubscribe without recompiling. The system can't adapt to runtime conditions.

Violation

Tight Coupling

StockFeed knows about MobileNotifier, RiskEngine, TaxReporter… Testing any of them in isolation requires the entire graph to exist.

The Core Insight

The Subject (StockFeed) should not know who is listening. It should only know that someone might be, and publish a notification. Observers register themselves — the subject never imports them. Adding a new observer requires zero changes to the subject.

// step_03

Deep Dive

The Observer Pattern has two participants and one relationship. But the details — push vs pull, weak references, event filtering, thread safety — are where real-world implementations live or die.

Diagram 1 — Core Structure

The Subject maintains a list of Observers behind an interface. It calls notifyObservers() on price change — never knowing concrete types. Observers register themselves; the Subject just iterates the list.

Diagram 2 — Push vs Pull Model

There are two ways to send data to observers. The choice has real trade-offs in coupling, flexibility, and performance.

Diagram 3 — Notification Flow

When updatePrice() is called, the Subject iterates its observer list and calls onPriceUpdate() on each. From the Subject's perspective, it is calling one method on an interface — it never knows which concrete types are in the list.

// push_vs_pull_and_event_objects
Push Model
Subject pushes data in the call

The subject packages all relevant data into the method call: onPriceUpdate(String symbol, double price, double prevPrice). Observers receive everything whether they need it or not.

Simple Less Flexible
Event Object
Best of both worlds

Pass a PriceEvent object with all fields. Observers read only what they need. Adding new fields to the event is non-breaking — old observers simply ignore them.

Recommended Extensible
Pull Model
Observer fetches what it needs

Subject passes itself: onPriceUpdate(Subject src). Observer calls src.getPrice(). Maximum flexibility — observer decides what to read. Costs extra coupling to Subject type.

Maximum Control More Coupling
Typed Events
Event bus / pub-sub variant

Multiple event types (PriceEvent, TradingHaltEvent, CircuitBreakerEvent). Observers subscribe to specific event types. This is how Spring's ApplicationEventPublisher works.

Enterprise Decoupled
// observer_pitfalls
  1. Memory Leaks (the classic bug)If observers hold references to subjects, and subjects hold references to observers, neither can be garbage collected. Solution: use WeakReference<Observer> in the subject's list, or always call unsubscribe() when an observer is done.
  2. Update Order DependenceIf Observer B depends on Observer A having processed the event first, you have a hidden coupling. Observers should be independent. If order matters, use a priority queue or chain-of-responsibility instead.
  3. Thread SafetyIf updatePrice() is called from a market-data thread while another thread is adding/removing observers, you will get ConcurrentModificationException. Solution: synchronise the observer list, or use CopyOnWriteArrayList.
  4. Cascading UpdatesObserver A's update() modifies the Subject, triggering another notifyObservers(), which calls A again, causing infinite recursion. Guard with an "updating" flag or make subjects immutable during notification.
  5. Too Many Fine-Grained NotificationsIf every tick fires a notification, observers may be overwhelmed. Debounce by throttling: only notify if price changed by >0.1%, or at most once per 100ms.
The Dependency Inversion Principle — Again

StockFeed used to depend on AlertService, Dashboard, etc. — concrete classes. After the refactor, it depends only on StockObserver — an abstraction. Both the subject and observers depend on the interface, not on each other. This is why adding a new observer requires zero changes to StockFeed.

Refactored — Full Observer System

PriceEvent.java — Event Object (recommended approach)Java
/**
 * Immutable event object. Observers read only what they need.
 * New fields can be added without breaking existing observers.
 * Using a record (Java 16+) makes it immutable and concise.
 */
public record PriceEvent(
    String symbol,
    double newPrice,
    double previousPrice,
    long   timestamp
) {
    // Convenience method — commonly needed by multiple observers
    public double percentChange() {
        return ((newPrice - previousPrice) / previousPrice) * 100.0;
    }
}
StockObserver.java — The Observer InterfaceJava
/**
 * Every observer implements this single method.
 * The @FunctionalInterface annotation means lambdas can be used
 * as observers — no need to create a named class for simple cases.
 */
@FunctionalInterface
public interface StockObserver {
    void onPriceUpdate(PriceEvent event);
}
StockFeed.java — Refactored SubjectJava
import java.util.*;
import java.util.concurrent.*;

public class StockFeed {

    private final String  symbol;
    private volatile double price;
    private volatile double previousPrice;

    // CopyOnWriteArrayList: thread-safe iteration without locking
    // Safe to add/remove observers while notification is in progress
    private final List<StockObserver> observers =
        new CopyOnWriteArrayList<>();

    public StockFeed(String symbol, double initialPrice) {
        this.symbol        = symbol;
        this.price         = initialPrice;
        this.previousPrice = initialPrice;
    }

    // ── Observer management ──────────────────────────────

    public void subscribe(StockObserver observer) {
        observers.add(observer);
    }

    public void unsubscribe(StockObserver observer) {
        observers.remove(observer);
    }

    // ── Price update ─────────────────────────────────────

    public void updatePrice(double newPrice) {
        previousPrice = price;
        price         = newPrice;
        notifyObservers();
    }

    private void notifyObservers() {
        PriceEvent event = new PriceEvent(
            symbol, price, previousPrice, System.currentTimeMillis()
        );
        // StockFeed never imports AlertService, Dashboard etc.
        // It only knows about StockObserver (the interface).
        observers.forEach(o -> o.onPriceUpdate(event));
    }

    public double getPrice()       { return price; }
    public String getSymbol()      { return symbol; }
    public int    observerCount()  { return observers.size(); }
}

ANALYST TIER
Challenges 1–3 · Core mechanics, first implementations
AnalystChallenge 1

Add a MobileNotifier Observer

Create a MobileNotifier that implements StockObserver. It should fire a push notification (print to console) only when the absolute percentage change exceeds a configurable threshold (e.g. 3%). Accept the threshold in the constructor. Subscribe it to two different StockFeed instances and confirm it only fires on large moves.

Goal

Write your first observer from scratch. Notice how StockFeed requires zero changes — you just implement the interface and subscribe. This is Open/Closed in action.

AnalystChallenge 2

Multi-Symbol Portfolio Feed

Create a Portfolio class that subscribes the same Dashboard observer to multiple StockFeed instances (AAPL, TSLA, NVDA). When any feed updates, the dashboard shows the price. The PriceEvent already carries symbol, so the same observer instance handles all three cleanly.

Think About

A single observer instance subscribed to N subjects is perfectly valid. The PriceEvent.symbol() field tells it which feed fired. This is why the event object pattern beats the method-signature approach.

AnalystChallenge 3

One-Shot Observer (Auto-Unsubscribe)

Create a OneShotAlert observer that fires exactly once — when the price crosses a target — then unsubscribes itself. The caller should receive a callback (pass a Runnable or Consumer<PriceEvent>) that fires on the trigger, then the observer removes itself from the feed so it never fires again.

Think About

The observer needs a reference to the StockFeed it is subscribed to in order to call unsubscribe(this). How do you pass that reference in cleanly? Constructor injection is the right answer.

QUANT TIER
Challenges 4–7 · Composition, filtering, thread safety
QuantChallenge 4

Filtered Observer Decorator

Create a FilteredObserver wrapper that takes any StockObserver and a Predicate<PriceEvent>. It only forwards events to the inner observer when the predicate returns true. Use it to give AlertService a "only fire between 09:30–16:00 market hours" filter without modifying AlertService at all.

Why This Matters

This is the Decorator pattern applied to observers — same structure you saw in the Command workshop. Notice how the same cross-pattern composition keeps appearing. FilteredObserver can wrap any observer and any predicate with zero changes to either.

QuantChallenge 5

Moving Average Observer

Create a MovingAverageObserver that maintains a sliding window of the last N prices (configurable, e.g. 5). Each time a new price arrives it updates the window and prints the current average. When the latest price crosses above or below the moving average, print a "CROSSOVER" signal.

Think About

The observer holds its own internal state between notifications. This is perfectly valid — each observer manages its own logic independently. The Subject has no idea this state exists.

QuantChallenge 6

Thread-Safe Observer List

The baseline uses CopyOnWriteArrayList. For a high-frequency feed, that copy-on-every-write cost matters. Refactor to use synchronized blocks on the observer list with a snapshot approach: copy the list under lock, then iterate the snapshot outside the lock. Write a test that adds an observer from Thread A while Thread B is mid-notification, and verify no ConcurrentModificationException.

Think About

What is the trade-off between CopyOnWriteArrayList (copy-on-write, ideal for read-heavy), synchronized snapshot (lower memory, more control), and ConcurrentLinkedQueue (lock-free)? Which wins in a market-data context where reads hugely outnumber writes?

QuantChallenge 7

Event Type Routing — Typed Event Bus

Add a second event type: TradingHaltEvent(String symbol, String reason). Create a MarketEventBus that lets observers subscribe to specific event types: bus.subscribe(TradingHaltEvent.class, myObserver). When an event is published, only observers registered for that type receive it. This mirrors Spring's ApplicationEventPublisher.

Think About

Use Map<Class<?>, List<Consumer<?>>> as the internal registry. How do you handle the cast safely? What happens if someone publishes an event type no one is listening to?

PORTFOLIO MANAGER TIER
Challenges 8–10 · Architecture, async, cross-pattern
PMChallenge 8

Async Observer with ExecutorService

Some observers are slow (e.g. AuditLogger writes to disk, RiskEngine runs a heavy calculation). Making StockFeed wait for them slows the price feed. Wrap the notification loop so each observer is called on a thread pool: executor.submit(() -> observer.onPriceUpdate(event)). Ensure the executor is properly shut down. Discuss the trade-off: what guarantees does synchronous notification give that async loses?

Think About

With async delivery, two events could be processed out of order by the same observer. A slow observer might process event N before finishing event N-1. How do you guarantee ordering per observer if you need it?

PMChallenge 9

Weak Reference Observer List

Simulate a memory leak: create 1000 short-lived observer objects, subscribe them without ever unsubscribing, then lose all references to them. Verify with a heap dump that they cannot be garbage collected because StockFeed holds strong references. Fix by storing WeakReference<StockObserver> in the list. Re-run and confirm the observers are collected. Add cleanup to remove dead references before each notification.

Why This Matters

This is one of the most common real-world Observer bugs. Android's LiveData, Java Swing listeners, and many event buses have all shipped with this bug at some point. Understanding it is a genuine senior-developer skill.

BOSSChallenge 10

New Domain — Reactive Stream (RxJava-style)

Build a minimal Observable<T> class with subscribe(Consumer<T>), map(Function<T,R>), filter(Predicate<T>), and throttle(long millis). Chain them: stockFeed.asObservable().filter(e -> e.percentChange() > 2.0).map(PriceEvent::symbol).subscribe(System.out::println). This is the skeleton of RxJava's Observable and Java 9's Flow API.

Why This Matters

RxJava, Project Reactor, and Java 9 Flow are all Observer Pattern at their core — with map/filter/flatMap operators added on top. Building this from scratch is the clearest possible demonstration that you understand the pattern at a deep level.

AnalystSolution 3 — One-Shot Alert

Observer that removes itself after firing once.

OneShotAlert.javaJava
import java.util.function.*;

public class OneShotAlert implements StockObserver {

    private final StockFeed            feed;
    private final Predicate<PriceEvent> trigger;
    private final Consumer<PriceEvent>  callback;

    public OneShotAlert(
        StockFeed            feed,
        Predicate<PriceEvent> trigger,
        Consumer<PriceEvent>  callback
    ) {
        this.feed     = feed;
        this.trigger  = trigger;
        this.callback = callback;
    }

    @Override
    public void onPriceUpdate(PriceEvent e) {
        if (trigger.test(e)) {
            callback.accept(e);
            feed.unsubscribe(this); // removes itself — fires exactly once
        }
    }
}

// Usage:
// OneShotAlert breakout = new OneShotAlert(
//   aapl,
//   e -> e.newPrice() > 200.0,           // trigger condition
//   e -> System.out.println("BREAKOUT!")  // callback
// );
// aapl.subscribe(breakout);
//
// aapl.updatePrice(195.0); // no trigger
// aapl.updatePrice(201.0); // triggers → fires → unsubscribes itself ✓
// aapl.updatePrice(205.0); // breakout no longer subscribed — silent ✓
QuantSolution 4 — FilteredObserver

Decorator pattern applied to observers.

FilteredObserver.javaJava
import java.util.function.*;

public class FilteredObserver implements StockObserver {

    private final StockObserver        inner;
    private final Predicate<PriceEvent> filter;

    public FilteredObserver(StockObserver inner, Predicate<PriceEvent> filter) {
        this.inner  = inner;
        this.filter = filter;
    }

    @Override
    public void onPriceUpdate(PriceEvent e) {
        if (filter.test(e)) {
            inner.onPriceUpdate(e); // only forward if predicate passes
        }
    }
}

QuantSolution 5 — Moving Average

Stateful observer — internal window, crossover signal.

MovingAverageObserver.javaJava
import java.util.*;

public class MovingAverageObserver implements StockObserver {

    private final int          windowSize;
    private final Deque<Double> window = new ArrayDeque<>();
    private double prevAvg = -1;

    public MovingAverageObserver(int windowSize) {
        this.windowSize = windowSize;
    }

    @Override
    public void onPriceUpdate(PriceEvent e) {
        window.addLast(e.newPrice());
        if (window.size() > windowSize) window.removeFirst();

        double avg = window.stream()
            .mapToDouble(Double::doubleValue)
            .average()
            .orElse(0);

        System.out.printf("[MA%d] %s price=%.2f avg=%.2f%n",
            windowSize, e.symbol(), e.newPrice(), avg);

        // Crossover signal
        if (prevAvg > 0) {
            boolean crossedAbove = e.previousPrice() < prevAvg && e.newPrice() >= avg;
            boolean crossedBelow = e.previousPrice() > prevAvg && e.newPrice() <= avg;
            if (crossedAbove) System.out.println("  ↑ CROSSOVER: price above MA");
            if (crossedBelow) System.out.println("  ↓ CROSSOVER: price below MA");
        }
        prevAvg = avg;
    }
}

// The Subject never knows this observer has internal state.
// Each observer manages its own logic independently. ✓
QuantSolution 6 — Thread-Safe Observer List

Snapshot under lock, iterate outside lock.

StockFeed.java — synchronized snapshot approachJava
private final List<StockObserver> observers = new ArrayList<>();
private final Object lock = new Object();

public void subscribe(StockObserver o) {
    synchronized (lock) { observers.add(o); }
}

public void unsubscribe(StockObserver o) {
    synchronized (lock) { observers.remove(o); }
}

private void notifyObservers() {
    List<StockObserver> snapshot;
    synchronized (lock) {
        snapshot = new ArrayList<>(observers); // copy under lock
    }
    // Iterate OUTSIDE the lock — observers can add/remove freely
    // without deadlock, even if onPriceUpdate() calls unsubscribe()
    PriceEvent event = new PriceEvent(symbol, price, previousPrice,
        System.currentTimeMillis());
    snapshot.forEach(o -> o.onPriceUpdate(event));
}

// Trade-offs:
// CopyOnWriteArrayList: copy on EVERY add/remove (write-heavy = expensive)
//   ideal when reads >> writes (read during notification vs write on subscribe)
//
// Synchronized snapshot: copy on EVERY notification (notification-heavy = expensive)
//   ideal when subscribe/unsubscribe >> notifications
//
// For a market feed: notifications are very frequent, subscribe/unsubscribe rare
//   → CopyOnWriteArrayList wins here
//
// For a UI event system: subscribe once, notify many times
//   → CopyOnWriteArrayList also wins
// The synchronized snapshot is better when notification rate is low.
QuantSolution 7 — Typed Event Bus

Map of event type → list of consumers. Spring-style.

MarketEventBus.javaJava
import java.util.*;
import java.util.function.*;

public class MarketEventBus {

    private final Map<Class<?>, List<Consumer<?>>> registry = new HashMap<>();

    public <T> void subscribe(Class<T> eventType, Consumer<T> handler) {
        registry.computeIfAbsent(eventType, k -> new ArrayList<>())
                .add(handler);
    }

    @SuppressWarnings("unchecked")
    public <T> void publish(T event) {
        List<Consumer<?>> handlers = registry.getOrDefault(
            event.getClass(), Collections.emptyList());
        handlers.forEach(h -> ((Consumer<T>) h).accept(event));
    }
}

PMSolution 8 — Async Observer

Non-blocking notification — feed doesn't wait for slow observers.

StockFeed.java — async notificationJava
private final ExecutorService executor =
    Executors.newCachedThreadPool();

private void notifyObserversAsync() {
    PriceEvent event = new PriceEvent(
        symbol, price, previousPrice, System.currentTimeMillis());

    observers.forEach(o ->
        executor.submit(() -> {
            try {
                o.onPriceUpdate(event);
            } catch (Exception ex) {
                System.err.println("Observer error: " + ex.getMessage());
            }
        })
    );
}

public void shutdown() {
    executor.shutdown();
    try { executor.awaitTermination(5, TimeUnit.SECONDS); }
    catch (InterruptedException e) { Thread.currentThread().interrupt(); }
}

// Trade-offs of async delivery:
//   SYNC: Simple, ordered, errors propagate to caller, easy to test
//     → Good for: UI events, in-process systems, low-frequency updates
//   ASYNC: Non-blocking, fast feed, fault-isolated observers
//     → Good for: market data feeds, IO-bound observers, high-frequency updates
//
// What async loses:
//   - Ordering guarantee: event N-1 may finish after event N
//   - Error visibility: observer exceptions don't surface to the caller
//   - "Happened-before" guarantees: harder to reason about state
//
// For ordering: use a single-thread executor per observer:
// Map perObserverExecutors;
// Each observer gets its own sequential queue. ✓
PMSolution 9 — Weak Reference Observer List

Prevent memory leaks — collect dead observers automatically.

StockFeed.java — weak reference listJava
import java.lang.ref.*;
import java.util.*;

private final List<WeakReference<StockObserver>> observers =
    new CopyOnWriteArrayList<>();

public void subscribe(StockObserver o) {
    observers.add(new WeakReference<>(o));
}

private void notifyObservers() {
    PriceEvent event = new PriceEvent(
        symbol, price, previousPrice, System.currentTimeMillis());

    var dead = new ArrayList<WeakReference<StockObserver>>();

    for (var ref : observers) {
        StockObserver o = ref.get(); // null if GC'd
        if (o != null) {
            o.onPriceUpdate(event);
        } else {
            dead.add(ref); // mark stale reference for removal
        }
    }
    observers.removeAll(dead); // purge collected observers
}

BOSSSolution 10 — Reactive Observable

RxJava skeleton: filter, map, throttle — all built on Observer.

Observable.java — minimal reactive streamJava
import java.util.*;
import java.util.function.*;

public class Observable<T> {

    private final List<Consumer<T>> subscribers = new ArrayList<>();

    public void subscribe(Consumer<T> subscriber) {
        subscribers.add(subscriber);
    }

    public void emit(T value) {
        subscribers.forEach(s -> s.accept(value));
    }

    // map: transform each item — returns NEW Observable of type R
    public <R> Observable<R> map(Function<T, R> mapper) {
        Observable<R> downstream = new Observable<>();
        this.subscribe(value -> downstream.emit(mapper.apply(value)));
        return downstream;
    }

    // filter: only forward values that pass the predicate
    public Observable<T> filter(Predicate<T> predicate) {
        Observable<T> downstream = new Observable<>();
        this.subscribe(value -> {
            if (predicate.test(value)) downstream.emit(value);
        });
        return downstream;
    }

    // throttle: only forward if at least 'millis' have passed since last emit
    public Observable<T> throttle(long millis) {
        Observable<T> downstream = new Observable<>();
        long[] lastEmit = { 0L };
        this.subscribe(value -> {
            long now = System.currentTimeMillis();
            if (now - lastEmit[0] >= millis) {
                lastEmit[0] = now;
                downstream.emit(value);
            }
        });
        return downstream;
    }
}

Real-World Connections

Java / Spring: ApplicationEventPublisher is a typed event bus (Solution 7). Spring's @EventListener annotation registers observers automatically via reflection.

RxJava / Project Reactor: Entire reactive programming model is Observer Pattern + operators. Observable, Flux, Mono are all Subjects. Subscribers are Observers. map/filter/flatMap are chained Observables (Solution 10).

Java 9 Flow API: Flow.Publisher = Subject. Flow.Subscriber = Observer. Flow.Subscription adds backpressure — the observer tells the subject how many items it can handle. This solves the "too many notifications" pitfall.

Android LiveData / StateFlow: Lifecycle-aware observers. Automatically unsubscribe when the UI component is destroyed — solving the memory leak problem (Solution 9) at the framework level.

Bloomberg / Reuters Market Data: Real financial market data infrastructure runs on Observer at massive scale. TIBCO Rendezvous, Solace, and Kafka are all distributed Observer systems where feeds are topics and subscribers are consumer groups.
