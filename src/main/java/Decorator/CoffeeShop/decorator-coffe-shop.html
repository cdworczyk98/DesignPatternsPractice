// Step 01

The Baseline App — No Pattern

A simple coffee shop ordering system. Works fine for now. No patterns needed yet — just plain Java classes with inheritance.

Domain

Coffee Shop

We sell beverages. Each beverage has a description and a cost. Customers can order: Espresso, DarkRoast, or HouseBlend.

Current Design

Simple Inheritance

One abstract Beverage base class. Each coffee type extends it and overrides cost() and getDescription().

Beverage.java Java 
// Abstract base class for all beverages
public abstract class Beverage {
    protected String description;

    public String getDescription() {
        return description;
    }

    public abstract double cost();
}
Espresso.java Java 
public class Espresso extends Beverage {

    public Espresso() {
        description = "Espresso";
    }

    @Override
    public double cost() {
        return 1.99;
    }
}
DarkRoast.java Java 
public class DarkRoast extends Beverage {

    public DarkRoast() {
        description = "Dark Roast";
    }

    @Override
    public double cost() {
        return 1.49;
    }
}
HouseBlend.java Java 
public class HouseBlend extends Beverage {

    public HouseBlend() {
        description = "House Blend";
    }

    @Override
    public double cost() {
        return 0.89;
    }
}
CoffeeShop.java Java — Main 
public class CoffeeShop {
    public static void main(String[] args) {
        Beverage b1 = new Espresso();
        Beverage b2 = new DarkRoast();
        Beverage b3 = new HouseBlend();

        System.out.println(b1.getDescription() + " $" + b1.cost());
        System.out.println(b2.getDescription() + " $" + b2.cost());
        System.out.println(b3.getDescription() + " $" + b3.cost());
    }
}
// Output:
// Espresso $1.99
// Dark Roast $1.49
// House Blend $0.89
✓ This Works

Clean, simple, readable. No over-engineering. This is the right design — until requirements change. Click "② The Problem" to see what breaks it.

// Step 02

New Requirements That Break It

Business is booming. Customers now want to add condiments — milk, soy, mocha, and whipped cream — each with its own cost. Your naive solution creates a class explosion.

⚠ New Requirement

Support any combination of condiments per beverage: Milk (+$0.25), Soy (+$0.20), Mocha (+$0.30), Whip (+$0.35). A customer should be able to order a "Double Mocha Soy Latte with Whip".

Anti-Pattern

The Naive Approach — Subclass Explosion

You create a subclass for every combination. With 3 beverages × 16 condiment combos, you need 48+ classes.

❌ Class Explosion (Don't do this) Java 
// Just a sample — imagine doing this for ALL combinations
public class EspressoWithMilk        extends Beverage { ... }
public class EspressoWithMilkAndMocha extends Beverage { ... }
public class EspressoWithSoy         extends Beverage { ... }
public class DarkRoastWithWhip       extends Beverage { ... }
public class DarkRoastWithMilkAndWhip extends Beverage { ... }
// ... 40+ more classes ...
Slightly Better, Still Bad

Boolean Flags in the Base Class

Put condiment flags directly in Beverage. Avoids class explosion but violates Open/Closed Principle — every new condiment requires editing the base class.

❌ Beverage.java (Flag-based anti-pattern) Java 
public abstract class Beverage {
    private boolean milk, soy, mocha, whip; // ← grows forever
    private double milkCost = 0.25, soyCost = 0.20;
    private double mochaCost = 0.30, whipCost = 0.35;

    public double condimentCost() {
        double total = 0;
        if (hasMilk())  total += milkCost;
        if (hasSoy())   total += soyCost;
        if (hasMocha()) total += mochaCost;
        if (hasWhip())  total += whipCost;
        return total;
    }
    // Problems:
    // 1. What if we add "oat milk"? Edit this class.
    // 2. What about double mocha? No support.
    // 3. Tea shouldn't have whip — but base class allows it.
    // 4. Violates Open/Closed Principle.
}
// Design Principles Violated
Violation

Open/Closed Principle

Adding a new condiment requires modifying the Beverage base class. Classes should be open for extension, closed for modification.

Violation

Single Responsibility

The base class now manages both beverage identity AND all possible condiment combinations. Too many reasons to change.

💡 The Key Insight

We need to add behavior to objects at runtime, not at compile time via subclassing. This is exactly what the Decorator Pattern solves.

// Step 03

The Decorator Pattern Explained

Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.

«interface» Component
▲                 ▲
ConcreteComponent      AbstractDecorator
▲
ConcreteDecoratorA   ConcreteDecoratorB
AbstractDecorator wraps a Component reference — that's the key
Role

Component (Interface)

Defines the interface for objects that can have responsibilities added. Both real objects and decorators implement this.

Role

Concrete Component

The real object being decorated. E.g. Espresso. It knows nothing about its decorators.

Role

Abstract Decorator

Implements the Component interface and holds a reference to a wrapped Component. The magic is in this composition.

Role

Concrete Decorator

Extends the abstract decorator. Adds its behavior before or after delegating to the wrapped component.

// How Wrapping Works

Nesting Call Chain

Each decorator delegates to the inner object. The chain unwinds like nested function calls:

Call Chain Visualization Pseudocode 
// Building the object:
Beverage order = new Whip(new Mocha(new Mocha(new DarkRoast())));

// Calling order.cost() triggers:
Whip.cost()
  → 0.35 + Mocha.cost()
        → 0.30 + Mocha.cost()
                → 0.30 + DarkRoast.cost()
                        → 1.49
                = 1.79
        = 2.09
  = 2.44  ← final result ✓
📐 Design Principle

Favor composition over inheritance. Each decorator IS-A Beverage (via interface) and HAS-A Beverage (via composition). This dual relationship is the pattern's power.

// Step 04

Refactored with Decorator Pattern

Here is the complete, clean refactor. Compare it carefully to the baseline version on panel ①.

Beverage.java — Component (unchanged) Java 
public abstract class Beverage {
    protected String description = "Unknown Beverage";

    public String getDescription() {
        return description;
    }

    public abstract double cost();
}
CondimentDecorator.java — Abstract Decorator Java 
// KEY: extends Beverage (IS-A) AND holds a Beverage (HAS-A)
public abstract class CondimentDecorator extends Beverage {
    protected Beverage beverage; // ← the wrapped component

    public CondimentDecorator(Beverage beverage) {
        this.beverage = beverage;
    }

    // Subclasses MUST override getDescription()
    public abstract String getDescription();
}
Mocha.java — Concrete Decorator Java 
public class Mocha extends CondimentDecorator {

    public Mocha(Beverage beverage) {
        super(beverage);
    }

    @Override
    public String getDescription() {
        return beverage.getDescription() + ", Mocha";
    }

    @Override
    public double cost() {
        return beverage.cost() + 0.30;
    }
}
Milk.java / Soy.java / Whip.java — Same pattern Java 
public class Milk extends CondimentDecorator {
    public Milk(Beverage b) { super(b); }

    @Override public String getDescription() {
        return beverage.getDescription() + ", Milk";
    }
    @Override public double cost() { return beverage.cost() + 0.25; }
}

public class Soy extends CondimentDecorator {
    public Soy(Beverage b) { super(b); }

    @Override public String getDescription() {
        return beverage.getDescription() + ", Soy";
    }
    @Override public double cost() { return beverage.cost() + 0.20; }
}

public class Whip extends CondimentDecorator {
    public Whip(Beverage b) { super(b); }

    @Override public String getDescription() {
        return beverage.getDescription() + ", Whip";
    }
    @Override public double cost() { return beverage.cost() + 0.35; }
}
CoffeeShop.java — Main (refactored) Java 
public class CoffeeShop {
    public static void main(String[] args) {

        // Plain espresso — no decorators
        Beverage b1 = new Espresso();
        print(b1);

        // Dark Roast + double Mocha + Whip
        Beverage b2 = new DarkRoast();
        b2 = new Mocha(b2);
        b2 = new Mocha(b2);  // double mocha!
        b2 = new Whip(b2);
        print(b2);

        // House Blend + Soy + Milk + Whip (one-liner style)
        Beverage b3 = new Whip(new Milk(new Soy(new HouseBlend())));
        print(b3);
    }

    static void print(Beverage b) {
        System.out.printf("%-45s $%.2f%n", b.getDescription(), b.cost());
    }
}

// Output:
// Espresso                                       $1.99
// Dark Roast, Mocha, Mocha, Whip                 $2.44
// House Blend, Soy, Milk, Whip                   $1.69
✓ What We Gained

Open/Closed: Add new condiment? Create one new class, touch nothing else.

Runtime flexibility: Combinations assembled dynamically, not baked into class hierarchy.

Double-ups work: new Mocha(new Mocha(...)) just works naturally.

// Step 05

Practice Challenges

Work through these in order. Each builds on the refactored coffee shop codebase from panel ④.

Easy Challenge 1

Add a New Condiment

Add an OatMilk condiment decorator that costs $0.45. Then verify you can order a "Espresso, Oat Milk, Mocha" and get the correct total ($2.74).

Hint

You should only need to create one new file. If you're editing Beverage.java or CondimentDecorator.java, you're doing it wrong.

Easy Challenge 2

Add a New Beverage

Add a Decaf beverage at $1.05. Confirm that all existing condiment decorators work on it without any changes to the decorator classes.

Goal

This demonstrates the Open/Closed Principle from the other direction: new concrete components slot in without touching decorators.

Medium Challenge 3

Size Multiplier Decorator

Add a SizeUpgrade decorator with an enum: SMALL (0.8×), MEDIUM (1.0×), LARGE (1.3×). It should modify the total cost by the multiplier and prepend the size to the description. A large double mocha dark roast should cost ~$3.17.

Think About

Should SizeUpgrade wrap the innermost or outermost object? Does the order matter? What are the pros/cons of each placement?

Medium Challenge 4

Duplicate Condiment Detection

Add a utility method countCondiment(Beverage b, Class<?> type) that walks the decorator chain and counts how many times a specific condiment appears. Use this to warn if a customer tries to add 3+ of the same condiment.

Think About

You'll need to use instanceof or reflection to inspect the chain. Should this logic live in the decorator, the beverage, or a separate utility class?

Hard Challenge 5

Loyalty Points Decorator

Extend Beverage with a getPoints() method (1 point per $0.50 spent, rounded down). Add a LoyaltyCard decorator that doubles the points earned. Design this without breaking the existing decorator chain.

Design Challenge

Adding getPoints() to the abstract base class is one approach, but requires updating all existing decorators. Can you use a separate interface to avoid this? Consider Pointable as an interface and think about the trade-offs.

Hard Challenge 6 — Open-Ended

Apply to a New Domain

Redesign the Decorator pattern for a text formatter system. Start with a PlainText component. Build decorators: BoldDecorator, ItalicDecorator, UpperCaseDecorator, TrimDecorator. The chain new Bold(new Italic(new Trim(new PlainText(" hello world ")))) should produce "<b><i>hello world</i></b>".

Stretch Goal

This is how Java's own InputStream / BufferedReader / InputStreamReader works. After finishing, read the Java I/O source — you'll recognize the pattern immediately.
