The Baseline App

An RPG character system with three warrior classes. Each character has a name, attack power, defense, and a describeSelf() method. Intermediate complexity — multiple fields, an interface, and a factory method already in place.

Domain

RPG Character Stats

Warriors, Mages, and Rogues each have different base stats. Characters can describe themselves. This is working, well-structured code.

Current Design

Interface + Abstract Class

A Character interface, an AbstractCharacter base, and three concrete classes. A CharacterFactory handles creation. Feels solid — until new requirements hit.

AbstractCharacter.java Java — Abstract Base 
public abstract class AbstractCharacter implements Character {

    protected final String name;
    protected int attack;
    protected int defense;
    protected int speed;

    public AbstractCharacter(String name, int attack, int defense, int speed) {
        this.name    = name;
        this.attack  = attack;
        this.defense = defense;
        this.speed   = speed;
    }

    @Override public String getName()    { return name; }
    @Override public int    getAttack()  { return attack; }
    @Override public int    getDefense() { return defense; }
    @Override public int    getSpeed()   { return speed; }

    @Override
    public String describeSelf() {
        return String.format(
            "[%s] ATK:%d  DEF:%d  SPD:%d",
            name, attack, defense, speed
        );
    }
}

// Step 02

New Requirements That Break It

The game design team adds an equipment system. Characters can now equip items — Sword, Shield, FireRune, ShadowCloak, SpeedBoots — each modifying stats and the description. A character can equip multiple items.

⚠ New Requirement

Equipment stacks on characters dynamically. "Thor with a Sword and Shield" has different stats than "Thor with a FireRune". Items can be equipped in any combination. New items will be added every sprint.

Anti-Pattern

Naive Subclass Approach

Create a subclass per combination. With 3 classes × 5 items × stacking = hundreds of classes. This gets worse with every new item added.

❌ Class explosion — don't do this Java 
public class WarriorWithSword            extends Warrior { ... }
public class WarriorWithSwordAndShield   extends Warrior { ... }
public class WarriorWithFireRuneAndSword extends Warrior { ... }
public class MageWithFireRune            extends Mage    { ... }
// ... 200+ more ...
Slightly Better, Still Bad

Equipment List in AbstractCharacter

Store a List<String> of equipment names and bake cost/stat logic into the base class via if chains. Works for 3 items; collapses for 20.

❌ AbstractCharacter.java — flag-based anti-pattern Java 
public abstract class AbstractCharacter implements Character {
    private boolean hasSword, hasShield, hasFireRune, hasShadowCloak;
    // ↑ grows forever with every new item

    public int getAttack() {
        int bonus = 0;
        if (hasSword)    bonus += 8;
        if (hasFireRune) bonus += 12;
        // add a new item? edit THIS method.
        return attack + bonus;
    }

    // Problems:
    // 1. Add SpeedBoots? Edit 4+ methods in this class.
    // 2. "Double Sword"? No support.
    // 3. A Mage equipping SpeedBoots overrides Warrior logic too.
    // 4. Violates Open/Closed. Violates Single Responsibility.
    // 5. Impossible to unit test items in isolation.
}
Violation

Open/Closed Principle

Every new equipment item requires opening and editing AbstractCharacter. A class should be closed for modification once stable.

Violation

Single Responsibility

The base class now owns base stats, class identity, AND the stat effects of every piece of equipment ever created. Three jobs. Three reasons to change.

Violation

Testability

You can't unit test Sword's stat bonus in isolation — it's buried inside a method on AbstractCharacter behind a boolean flag.

Violation

Scalability

A real RPG might have 50+ items. Each one adds more booleans, more if branches, more methods to edit. The class becomes unmaintainable.

💡 The Key Insight

Equipment is behaviour added at runtime, not at compile time. We need a way to wrap a character in layers of equipment — each layer knowing only about the layer beneath it. That's exactly what Decorator does.

// Step 03

Deep Dive — How Decorator Works

This panel goes beyond the surface. We'll look at the structural mechanics, the mental model, where people go wrong, and the limits of the pattern.

The Two Relationships

Every decorator has exactly two relationships with the component type. Both are essential:

IS-A

Inheritance (Type)

The decorator extends or implements the same type as the component it wraps. This means you can pass a decorated object anywhere an undecorated one is expected. The client sees a Character — it never knows if it's wrapped or not.

HAS-A

Composition (Behaviour)

The decorator holds a reference to a wrapped component. It delegates all calls to the inner object, adding its own behaviour before or after the delegation. This is the actual mechanism of extension.

The Delegation Chain — Step by Step

When you call .getAttack() on a wrapped character, execution flows inward through each decorator:

Unwinding the call stackPseudocode
// Build: Thor equipped with Sword, then FireRune on top
Character thor = new FireRune(new Sword(new Warrior("Thor")));

// thor.getAttack() call chain:
FireRune.getAttack()
  → 12 + Sword.getAttack()
        → 8 + Warrior.getAttack()
                → 18      ← base stat
        = 26
  = 38  ← final result (18 base + 8 sword + 12 fire rune)

// thor.describeSelf() chain builds a string outward:
FireRune.describeSelf()
  → "[FireRune] " + Sword.describeSelf()
        → "[Sword] " + Warrior.describeSelf()
                → "⚔  Warrior | [Thor] ATK:38 DEF:15 SPD:8"
Why "Before or After" Matters

A decorator can add behaviour before delegating, after delegating, or both. The order changes the result — this is intentional power, not a bug:

Before vs After delegationJava
// Adding BEFORE: prefix the description
@Override
public String describeSelf() {
    return "[Sword +8ATK] " + character.describeSelf();
    // → "[Sword +8ATK] ⚔ Warrior | [Thor]..."
}

// Adding AFTER: append the description
@Override
public String describeSelf() {
    return character.describeSelf() + " + [Sword +8ATK]";
    // → "⚔ Warrior | [Thor]... + [Sword +8ATK]"
}

// Both: wrap with a logging call around delegation
@Override
public int getAttack() {
    System.out.println("Sword: applying +8 attack bonus"); // before
    int result = character.getAttack() + 8;
    System.out.println("Sword: result = " + result); // after
    return result;
}
Common Mistakes to Avoid

  1. Storing mutable state in decorators

    Decorators should be stateless transformations. If you find yourself adding private int charges = 3 to a decorator, you may need a different pattern (State or Strategy). Mutable decorator state causes confusion when the same decorator wraps different objects.

  2. Using concrete types instead of the interface

    The decorator's wrapped field must be typed as the interface (Character character), not a concrete class (Warrior character). Using a concrete type defeats the whole purpose — it means your decorator only works on one class.

  3. Forgetting to delegate

    Every method that the decorator doesn't explicitly modify must still delegate to the wrapped object. Forgetting to call character.getSpeed() in a Sword decorator means speed returns 0 — a silent bug that's hard to find.

  4. Applying Decorator when Builder fits better

    If the configuration is known at creation time and doesn't change, the Builder pattern is cleaner. Decorator shines for runtime additions. Ask yourself: is this equipment decided when the character is first built, or dynamically swapped during gameplay?

📐 Design Principle: Favour Composition Over Inheritance

Every OOP textbook says it. Decorator shows you why. Inheritance is a compile-time, static relationship. Composition is a runtime, dynamic relationship. Equipment on an RPG character is inherently dynamic — it changes as the player plays. Decorator maps perfectly to that reality.

Refactored with Decorator Pattern

The Character interface stays untouched. We add one abstract decorator and one class per equipment item. The existing Warrior, Mage, Rogue classes are not modified.

EquipmentDecorator.java — Abstract Decorator Java 
/**
 * IS-A Character (implements interface) so it can stand in for any Character.
 * HAS-A Character (holds reference) so it can delegate and extend behaviour.
 */
public abstract class EquipmentDecorator implements Character {

    protected final Character character; // ← typed as interface, not concrete class!

    public EquipmentDecorator(Character character) {
        this.character = character;
    }

    // Default delegation — subclasses override only what they modify
    @Override public String getName()    { return character.getName(); }
    @Override public int    getAttack()  { return character.getAttack(); }
    @Override public int    getDefense() { return character.getDefense(); }
    @Override public int    getSpeed()   { return character.getSpeed(); }
    @Override public String describeSelf() { return character.describeSelf(); }
}
🔑 The Default Delegation Pattern

By providing default delegation in the abstract class, concrete decorators only need to override the methods they actually change. Sword overrides getAttack() and describeSelf() — it doesn't touch getDefense() or getSpeed() at all. Less boilerplate, fewer bugs.

VeteranSolution 3 — Stat Inspector

Walk the chain with instanceof. Field visibility must be at least package-private.

CharacterInspector.javaJava
import java.util.*;

public class CharacterInspector {

    public static List<String> listEquipment(Character c) {
        List<String> equipment = new ArrayList<>();
        Character current = c;

        while (current instanceof EquipmentDecorator ed) {
            equipment.add(ed.getClass().getSimpleName());
            current = ed.character; // ← requires 'protected' or package visibility
        }
        return equipment; // outermost first
    }

    public static void printStats(Character c) {
        System.out.printf(
            "%-40s  ATK:%-3d DEF:%-3d SPD:%-3d%n",
            c.getName() + " [" + String.join(",", listEquipment(c)) + "]",
            c.getAttack(), c.getDefense(), c.getSpeed()
        );
    }
}

// Note: EquipmentDecorator.character field should be 'protected'
// (it already is in our design). Using the pattern variable 'ed'
// in the instanceof check is Java 16+ pattern matching syntax.
💡 Encapsulation Trade-off

Exposing character as protected lets utility classes inspect the chain — but only from within the same package or subclasses. If you need stricter encapsulation, add a getWrapped() method to EquipmentDecorator instead. That's the idiomatic Java approach.

VeteranSolution 4 — CursedBlade

Negative stat modifier with a 0-floor guard.

CursedBlade.javaJava
public class CursedBlade extends EquipmentDecorator {

    private static final int ATTACK_BONUS  =  20;
    private static final int DEFENSE_CURSE = -8;

    public CursedBlade(Character c) { super(c); }

    @Override
    public int getAttack() {
        return character.getAttack() + ATTACK_BONUS;
    }

    @Override
    public int getDefense() {
        // Guard lives HERE in the decorator — not in the base class.
        // Reason: the base class shouldn't know about item-specific
        // constraints. Each item owns its own rules.
        return Math.max(0, character.getDefense() + DEFENSE_CURSE);
    }

    @Override
    public String describeSelf() {
        return character.describeSelf() + " + [CursedBlade +20ATK -8DEF]";
    }
}

// Character mage = new CursedBlade(new Mage("Lyra"));
// mage.getDefense() → Math.max(0, 6 - 8) = 0  (not -2) ✓
VeteranSolution 5 — Unique Equipment Enforcement

Factory-style guard before wrapping.

EquipmentValidator.javaJava
public class EquipmentValidator {

    /**
     * Safely wraps 'base' with 'item', but throws if that
     * item type is already anywhere in the decorator chain.
     */
    public static Character equip(Character base, EquipmentDecorator item) {
        Class<?> itemType = item.getClass();
        Character current = base;

        while (current instanceof EquipmentDecorator ed) {
            if (ed.getClass() == itemType) {
                throw new IllegalStateException(
                    "Cannot equip " + itemType.getSimpleName() +
                    ": already equipped on " + base.getName()
                );
            }
            current = ed.character;
        }
        // Safe to wrap — item not already present
        item.character = base; // requires package-private or setter
        return item;
    }
}

// Better approach — pass the base to item constructor as usual,
// but validate before construction:
// EquipmentValidator.assertNotEquipped(base, SpeedBoots.class);
// Character c = new SpeedBoots(base); // only if check passed
💡 Design Note

A cleaner approach separates validation from construction: validate first, then use the normal new SpeedBoots(base) constructor. This avoids mutating the decorator after construction and keeps constructors pure.

EliteSolution 6 — Shadow Set Bonus

Detect chain composition at construction time.

ShadowSetBonus.javaJava
public class ShadowSetBonus extends EquipmentDecorator {

    private static final int SET_BONUS = 10;
    private final boolean bonusActive;

    public ShadowSetBonus(Character c) {
        super(c);
        // Check the chain ONCE at construction time — more efficient
        // than checking on every getSpeed() call
        this.bonusActive = hasItem(c, ShadowCloak.class)
                        && hasItem(c, SpeedBoots.class);
    }

    private static boolean hasItem(Character c, Class<?> type) {
        while (c instanceof EquipmentDecorator ed) {
            if (type.isInstance(ed)) return true;
            c = ed.character;
        }
        return false;
    }

    @Override
    public int getSpeed() {
        return character.getSpeed() + (bonusActive ? SET_BONUS : 0);
    }

    @Override
    public String describeSelf() {
        String bonus = bonusActive ? " + [SHADOW SET BONUS +10SPD]" : "";
        return character.describeSelf() + bonus;
    }
}

// Usage:
// Character zane = new ShadowSetBonus(
//     new SpeedBoots(new ShadowCloak(new Rogue("Zane")))
// );
// zane.getSpeed() → 20 (base) + 7 (cloak) + 15 (boots) + 10 (set) = 52 ✓
EliteSolution 7 — Build Code Serialisation

Each decorator appends its code; parser reverses the wrapping.

Character.java (updated) + Decorators (snippet)Java
// Add to Character interface with a default:
public interface Character {
    // ... existing methods ...
    default String getBuildCode() { return getName(); }
}

// In AbstractCharacter — base case
@Override
public String getBuildCode() { return name; }

// In Sword decorator:
@Override
public String getBuildCode() {
    return character.getBuildCode() + "|SW";
}
// FireRune → "|FR", Shield → "|SH",
// SpeedBoots → "|SB", ShadowCloak → "|SC"
BuildParser.javaJava
public class BuildParser {

    public static Character parse(String code) {
        String[] parts = code.split("\\|");
        // parts[0] = name (we'll default to Warrior for simplicity)
        Character c = new Warrior(parts[0]);

        // Apply equipment in order (index 1 onwards)
        for (int i = 1; i < parts.length; i++) {
            c = switch (parts[i]) {
                case "SW" -> new Sword(c);
                case "SH" -> new Shield(c);
                case "FR" -> new FireRune(c);
                case "SB" -> new SpeedBoots(c);
                case "SC" -> new ShadowCloak(c);
                default    -> throw new IllegalArgumentException("Unknown code: " + parts[i]);
            };
        }
        return c;
    }
}

// Round-trip test:
// Character original = new FireRune(new Sword(new Warrior("Thor")));
// String code = original.getBuildCode();   // → "Thor|SW|FR"
// Character restored = BuildParser.parse(code);
// restored.getAttack() → 18 + 8 + 12 = 38 ✓
BossSolution 8 — Middleware Pipeline

Same pattern, production-grade domain. Identical structure.

HttpRequest.java + RealRequest.java + AbstractMiddleware.javaJava
// Component interface
public interface HttpRequest {
    String execute(String token);
}

// Concrete component (the real work)
public class RealRequest implements HttpRequest {
    @Override
    public String execute(String token) {
        return "{ \"data\": \"secret payload\" }";
    }
}

// Abstract decorator
public abstract class AbstractMiddleware implements HttpRequest {
    protected final HttpRequest next;
    public AbstractMiddleware(HttpRequest next) { this.next = next; }
}
AuthMiddleware.java / LoggingMiddleware.java / CacheMiddleware.javaJava
public class AuthMiddleware extends AbstractMiddleware {
    private static final String VALID = "Bearer secret123";

    public AuthMiddleware(HttpRequest next) { super(next); }

    @Override
    public String execute(String token) {
        if (!VALID.equals(token))
            throw new SecurityException("401 Unauthorized");
        return next.execute(token); // add before, delegate
    }
}

public class LoggingMiddleware extends AbstractMiddleware {
    public LoggingMiddleware(HttpRequest next) { super(next); }

    @Override
    public String execute(String token) {
        System.out.println("[LOG] Request starting...");   // before
        String result = next.execute(token);
        System.out.println("[LOG] Response: " + result);    // after
        return result;
    }
}

public class CacheMiddleware extends AbstractMiddleware {
    private final Map<String, String> cache = new HashMap<>();

    public CacheMiddleware(HttpRequest next) { super(next); }

    @Override
    public String execute(String token) {
        if (cache.containsKey(token)) {
            System.out.println("[CACHE] Hit!");
            return cache.get(token); // short-circuit — no delegation
        }
        String result = next.execute(token);
        cache.put(token, result);  // store after delegation
        return result;
    }
}

🔗 Real-World Connection

This is exactly how Express.js app.use(middleware) works, how ASP.NET Core's IMiddleware pipeline works, and how Spring's OncePerRequestFilter chain works. The only difference is that those frameworks wire up the chain for you. The Decorator structure underneath is identical to what you just wrote.
